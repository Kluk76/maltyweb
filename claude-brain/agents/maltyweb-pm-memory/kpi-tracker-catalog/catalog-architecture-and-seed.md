## BLESSED ARCHITECTURE (PM verdict 2026-06-06)
**Model = each tracker is a THIN compute_handler wrapping an EXISTING canonical metric; `mon-tableau.php` + recap email share the handler. A tracker EXPOSES a fact, NEVER recomputes one a second way. This is the feature's cardinal rule.** Blessed. The risk is not the model ‚ÄĒ it's discipline-slippage in COGS/sales/WIP/packaging-consumption handlers.

### compute_handler taxonomy ‚ÄĒ HYBRID REGISTRY (recommended)
- NO 250 hardcoded PHP functions; NO free-SQL-in-DB column (unauditable/injectable/un-versionable ‚ÄĒ anti-pattern).
- **~15-25 PHP handler functions grouped by `source_domain`** (collapse 13 cats to ~12 source surfaces: wort/tanks/racking/packaging/fg_stock/sales/rm_procurement/qa/cogs/utilities/ops_health/equipment). One parameterized handler serves a whole category ‚ÄĒ trackers within a category differ only by aggregation axis + viz.
- **Per-tracker variation = DATA in a typed `params_json`** (period window, group-by axis, SKU/format filter, metric column, viz_type). Handler shapes its query from a **whitelisted vocabulary** (allowed group-by cols / metrics / filters, enum-validated in PHP). A tracker whose params reference a column outside the whitelist must REFUSE (surface to admin), not silently run. Same posture as `ref_pages.domain` ENUM gating render.
