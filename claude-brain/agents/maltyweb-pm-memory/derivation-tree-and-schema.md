- **Données générales** (system settings, `system_settings`/`ref_sites`): date format = jj/mm/aaaa DMY day-first system-wide, language, brewery name, sites, users. Ingestion reads it. Never hardcode these.
- **Salle des Machines** — PRODUCTION master-data root. The commissioning + capacity layer:

## SQL BUILD SCHEMA (canonical maltytask DB)
- **`schema_meta` classifies EVERY table** (since migration 080): `table_class` (reference/lookup/source/derived/audit/config/system), `corrections_policy` (allowed / allowed_with_side_effect / blocked / blocked_with_redirect), `writer_script`, `upstream_hint`. **Query it before any write.** Derived + blocked_with_redirect → fix upstream, not the cache.
- **FK type cheat sheet:** `ref_*.id` = INT UNSIGNED; `bd_*`/`doc_*`/`inv_*`/audit/event = BIGINT UNSIGNED. **Exception:** `ref_packaging_formats.id` = BIGINT UNSIGNED (drift — keep). FK columns must match target PK type exactly.
- **Migration discipline:** `db/migrations/NNN_*.sql`; MySQL 8.0.45 (NO `IF NOT EXISTS` on ALTER clauses; `CREATE TABLE IF NOT EXISTS` ok; CHECK names table-prefixed; CASCADE-FK column not in a CHECK; no bare SELECT in a migration — PDO exec leaves result sets open); every NEW table INSERTs its `schema_meta` row in the same migration; apply via `migrate.php` (auto-records `schema_migrations`, keys on `filename` not `version`); collation utf8mb4_unicode_ci on new text/enums. **`maltytask_app` LACKS CREATE VIEW privilege** (bit migration 144 — see volume-dimension.md; orchestrator fixing).
- **Write discipline:** `--dry-run` default for one-shot scripts; snapshot affected rows before write; idempotency (row_hash / content-key); `ON DUPLICATE KEY UPDATE` for upserts (not `INSERT IGNORE` when you meant to overwrite).
- **Common SQL traps** (see `sql` skill in full): ONLY_FULL_GROUP_BY → `ANY_VALUE()`; collation mismatch on JOIN (`bd_*` = utf8mb4_0900_ai_ci vs `ref_*`/`inv_*` = utf8mb4_unicode_ci) → explicit `COLLATE`; EUR→CHF `*IF(currency='EUR',0.945,1)`; `is_inventoried=1` filter on warehouse pages.
- (Applied-migrations chronology 131–144 is in build-history.md.)
- **MI MERGE / TOMBSTONE-AND-RELINK pattern (canonical since mig 254, El Dorado dup-MI merge 2026-06-01, PM-verified live).** When two `ref_mi` rows model ONE physical ingredient (here `HOPS_ELDORADO` id 572 vs `HOPS_EL_DORADO` id 34 — duplicate MI), the merge migration KEEPS the canonical id and **tombstones** the loser: set loser `is_active=0, is_inventoried=0` (NOT a row delete — preserve audit/FK integrity), then **repoint every FK that pointed at the loser to the keeper.** FK surfaces that had to move in mig 254 (= the canonical relink checklist for any MI merge): `inv_deliveries.ingredient_fk` (+ rewrite `mi_id_raw` string to the keeper's code so re-ingest resolves to the keeper — mig 254 set 'HOPS_ELDORADO'→'HOPS_EL_DORADO'), `inv_rm_stocktake.mi_id_fk`, `inv_rm_stocktake_lines.mi_id_fk`, `ref_mi_aliases.mi_id_fk` (move all the loser's supplier-code aliases onto the keeper — and an alias insert that case-insensitively matches an existing keeper alias is a NO-OP, do not create a dup, e.g. 'Eldorado' vs existing 'ElDorado'). Audit: one `audit_row_revisions` row per touched row (`comment LIKE 'migNNN_%'`) — **count fans out per alias** (mig 254 = 10 rows: 2 deliveries + 1 stocktake + 1 lines + 5 aliases + 1 tombstone, NOT 9). Wrap the whole DML body in `START TRANSACTION; … COMMIT;`. Post-merge `v_mi_wac` on the keeper correctly unions both suppliers' live deliveries (mig 254: keeper id 34 → delivery_rows=2, supplier_count=2, qty_remaining_total=205, wac_chf=13.85). Verification result (all PASS): loser tombstoned (active/inventoried both 0), ZERO pointers remain at the loser across all 5 FK surfaces + the raw string.
- **`is_inventoried` = the SINGLE semantic "counted-as-stock vs expensed-on-purchase" flag (CANONICAL RULE, operator-confirmed 2026-06-01).** It is **NOT derivable from `gl_account`** — live data splits many GLs across both states, so NO GL→inventoried derivation was built and none should be. `is_inventoried=0` = expensed on purchase, never counted in RM stocktake (drops out of the count dropdown gate + every RM read). - **⚠️ uniq_row_hash-after-repoint GOTCHA (mig 254, MUST-KNOW for any rollup-fed stocktake relink).** `inv_rm_stocktake.row_hash` is **NOT NULL + UNIQUE** — you cannot NULL it when you repoint `mi_id_fk`, and leaving the loser's old hash is stale/wrong-keyed. FIX = set a **deterministic per-row UNIQUE sentinel** that cannot collide and gets cleanly overwritten by the next rollup: `row_hash = SHA2(CONCAT('migNNN-relink-', id), 256)`. This works because `rm_recompute_rollup(mi_code, mi_id, period)` upserts on `(mi_id, period)` and **recomputes row_hash** — proven live: re-running `rm_recompute_rollup('HOPS_EL_DORADO',34,'2026-05')` reclaimed the relinked row (no 1062), `counted_qty` preserved (170kg), row_hash flipped sentinel(de84…)→recomputed(bf67…). **Contrast `inv_rm_stocktake_lines.row_hash`:** it's a nonce'd per-line hash that is NOT rollup-rewritten → leave it AS-IS on relink (only move `mi_id_fk`). Rule: a stocktake row whose hash IS the rollup's `(mi_id,period)` key gets the sentinel; a line whose hash is a stable nonce just gets the FK moved.
