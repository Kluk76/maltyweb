**CONFIRMED SEMANTICS (operator + maltyweb saisie, 2026-06-08):** A multi-brew batch = N parallel ~30 HL brews. The grain bill on the brewing-ingredients form is entered **PER SINGLE BREW** (each parallel brew uses that same charge). So `bd_brewing_ingredients_v2` = ONE per-brew record per batch; `bd_brewing_cooling` = N rows (one per brew). Triangulated on EMB: recette Pilsener 14.83 kg/HL; F2 wrongly 2.79; corrected 347 kg × 4 ÷ 124.5 = 11.15 kg/HL (≈ recette with real drift). **So 176 kg IS the per-brew charge** → Fix B confirmed (numerator × N_brews), NOT a denominator change.

**1. FIX MECHANISM — CANONICAL CHOICE = "numerator × N_brews where N_brews = COUNT(bd_brewing_cooling rows for the batch)".** This is the minimal, robust fix. Rejected the alternative ("denominator = batch_hl / N_brews = average single-brew volume") on edge-case robustness:
   - **Both are algebraically equivalent ONLY when all brews are equal-volume.** They DIVERGE on unequal brews and on a missing cooling row — and the divergence favors numerator×N.
   - **numerator × N** keeps the denominator as the TRUE total cooled volume (Σ real HL). The per-brew bill is physically repeated N times; multiplying by the count of actually-cooled brews is the faithful model of "this charge, brewed N times, into this much total beer". Unequal brews are handled correctly: the per-HL is (per-brew-qty × N) ÷ Σ(actual HL) = exactly the malt-mass-over-total-volume that COGS wants.
   - **denominator/N (average single-brew volume)** throws away the real per-brew volume distribution and re-introduces an averaging assumption; on a batch with one missing/zeroed cooling row it would compute the average over the SURVIVING rows but still divide a single-brew numerator, silently re-creating a partial version of the same mismatch. Worse failure mode.
   - **Edge — missing cooling row:** with numerator×N, N = count of *surviving* (volume>0) cooling rows (the §2a query already filters `cool_final_volume_hl > 0`); numerator scales by the same N that built the denominator, so the two stay consistent even when a brew's volume is absent. A genuinely missing cooling row degrades gracefully (under-counts N AND HL together → per-HL stays ~right; mass attribution is the only loss, and that's a data-gap, not a basis error). This self-consistency is the decisive reason.
   - **Edge — unequal brew volumes:** numerator×N ÷ Σ(real HL) is correct by construction (it's literally total grain ÷ total volume). The average-volume variant is NOT.
   - **CANONICAL: numerator × N_brews, N_brews = the count of cooling rows that built `batch_hl` for that (recipe,batch).** Compute N in the SAME §2a GROUP BY that produces `batch_hl` (add `COUNT(*) AS n_brews` alongside `SUM(cool_final_volume_hl)`), carry it in a `$batchNBrews[$rid][$batch]` index parallel to `$batchHl`. Do NOT recount elsewhere — single source of N.

**2. VOLUME-WEIGHTING INTERACTION — × N_brews goes on the PER-BATCH NUMERATOR, BEFORE the MAD/per-HL computation. Weighting stays correct.** The aggregator (`$computePerHl`, §3b) builds per-batch per_hl = qty ÷ hl for MAD, then sums surviving qty ÷ ΣHL. **The fix is to scale `qty` to batch-total (× N_brews) at the point the per-batch data enters the aggregator — i.e. fold N_brews into the numerator qty as it's read into `$batchQty`, so BOTH the per-batch per_hl (MAD input) AND the final Σqty÷ΣHL see the batch-total mass.** Consequences, all correct:
   - Per-batch per_hl for MAD becomes `(qty×N) ÷ batch_hl` = true kg/HL for that batch → MAD now compares real per-HL values across batches (today it compares ÷N values, which are internally consistent so MAD still "worked", but the fix makes them physically meaningful — no MAD-behavior regression because the relative dispersion within a recipe is unchanged by a per-batch scalar ONLY IF N is constant across batches; **N can VARY per batch** (a 4-brew batch and a 2-brew batch of the same recipe), so the per-batch scaling MUST happen before MAD, not after — confirmed this is the right place).
   - Final per_hl = Σ(qty_i × N_i) ÷ Σ(batch_hl_i) across the basis window. A 4-brew batch contributes (per-brew-qty × 4) to the numerator and its ~124 HL to the denominator; a 1-brew batch contributes (per-brew-qty × 1) and ~30 HL. **This IS the correct volume-weighting: each batch weights by its true total HL, and each batch's mass is its true total mass.** A 4-brew batch correctly weighs ~4× a 1-brew batch. ✓
   - **× N_brews on the numerator is the right place — NOT a post-hoc scale of the final per_hl, NOT inside the windowing/selection.** Windowing (basis-batch selection, ≤8, DH-gate) operates on batch identity, unaffected. MAD must see the scaled per-batch value (because N varies per batch) → scale at numerator-read. Confirmed.
