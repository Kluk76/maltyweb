- **REFUSE-DON'T-NULL — precedent register (load-bearing patterns).** The discipline shows up in several places; carry the right pattern for the right axis:
  - **BOM line emission** (original): when a slot's MI can't be resolved, the recompute emits a self-sufficient ReviewQueue row, NEVER a `mi_id=NULL` line (would silently zero a COGS component).
  - **Compiler input-side / requested-SKU assertion** (NEW 2026-05-28, commit `de90755`, `app/sku-bom-compile.php` ~line 499): the compiler refuses to process a requested SKU AT FUNCTION ENTRY if its format is NOT commissioned (not in `_compiler_gated_format_ids()`) AND NOT composite. Use this shape when a build agent needs to gate a function on a precondition that, if violated, would otherwise produce silent zero/empty output.

- **"Raw text is the truth" — never speculate on identity mappings (re-confirmed 2026-05-28 by the BLO/BLA mis-routing audit).** When an event row carries BOTH a brewing-side resolved key (e.g. `recipe_id_fk`) AND a raw operator-typed identifier (e.g. `bd_packaging_v2.neb_beer = "BLO4C"`), the RAW text is the authoritative signal for any layer that the brewing-side key does NOT own — here, the packaging-SKU layer. White-label/relabel SKUs split LIQUID identity from PACKAGE identity by design; a lookup keyed only on `(recipe_id, format)` silently mis-routes (72 events / ~1,779 HL of BLO/BLA production landed on ZEP/MOO SKUs over 5 years). **Rule: when resolving a layer (SKU/MI/client/…), check the raw text FIRST, fall back to derived keys only when raw is missing or unrecognized.** The protection that did fire here was operator-surfacing: we showed the raw `neb_beer` to the operator and they confirmed the BLO/BLA semantics rather than letting the auditor infer from prices/patterns. → packaging-bom-model/README.md §THREE SKU IDENTITY LAYERS. Distinct from the gate-helper parity rule below (this is input-parsing; that is rule-consistency).
- **QUERY-CANONICAL-BEFORE-LABELING-IDENTITIES (NEW 2026-05-29, vessel-commissioning audit F2 closure — audit-side analogue of the "raw text is the truth" rule).** When an audit or a build labels a SKU, recipe, beer, client, supplier, or category that drives a flag value / triage verdict / scrapping decision, query the canonical table FIRST — never infer the identity from a code prefix, digit run, or pattern. The F2 closure caught two mid-arc speculation failures: (1) auditor labelled DGDB/DGDF "DGD prefix unfamiliar — possibly discontinued" without querying `ref_beer_types` (DGD = DrunkBeard "Galactic Drift" CollabIn, ingested same day into `ref_beer_types.id=56`, recipe `ref_recipes.id=31` — was in scope, was the canonical source); (2) auditor read EPH24PB / EPH44PB as "24-pack/44-pack" from the digit run instead of querying `ref_recipes` for the EPH vintage layout (they are EPH2-as-4PB and EPH4-as-4PB — standard 4PB form for EPH2 and EPH4 seasonals). The canonical sources to consult before labelling: `ref_beer_types` (beer/recipe identity + subtype Core/EPH/CollabIn/CollabOut/WhiteLabel/Archive — `project_beertypes_canonical_classifier`), `ref_recipes` (recipe id + sku_prefix + subtype + vintage layout for EPH), `ref_skus` (sku_code + format_id + recipe_id_fk), `ref_clients` / `ref_suppliers` (counterparty identity), `ref_packaging_formats` + `dbc_packaging_format_templates` (format vs container vs run_type). **Audit-axis discipline:** any audit verdict that turns on an identity claim must cite the canonical row it queried, not the code pattern it inferred from. Same family as the "never speculate on identity mappings" rule (which guards INGEST/resolution); this one guards AUDIT/labeling. The protection mechanism is identical: surface the raw identifier to the operator OR check the canonical table, never infer from pattern.
- **🔴 DO-NOT-REPEAT — a BACKFILL/RECOMPUTE MUST NOT auto-populate CONTRACT rows via the compute fn's generic fallback (recorded 2026-06-09, loss-model-consistency arc, `7a8b63a`).** When a fix recomputes a `bd_packaging_v2` (or any beer-tax/COGS-bearing) column across history, the compute fn's generic `$isContract` (or any "default/else") branch will happily populate rows that were DELIBERATELY left NULL — here, 57 previously-NULL contract rows (BLZ, Abbaye de St-Maurice, MeltingPote, Septentrion, Obrist…) got +858 HL of contract beer dumped into the tax base (true backfill total +1399 vs the dry-run's +554; the gap WAS the over-reach). Opus caught it ONLY via an independent snapshot diff (the agent's own gate passed). **RULE: a recompute/backfill restates rows that ALREADY had a non-NULL value; it must NOT newly populate rows that were NULL — scope the UPDATE with `WHERE <col> IS NOT NULL` (or an explicit row allowlist), and verify the row COUNT touched + the aggregate delta against an independent pre-image snapshot before commit. Contract packaging specifically needs per-client SKU mapping + a contract-beer-tax-liability decision (is Néb the taxpayer on contract beer?) BEFORE any value lands — never via a generic fallback.** This is the COGS-bearing-claim verification rule (line 7) applied to the BACKFILL axis: positivity/existence gates miss a too-large restatement; assert the row-set and the magnitude.
- **🟡 PARALLEL-SESSION COMMIT HYGIENE — always `git commit <explicit paths>`, never bare `git commit` after `git add` (near-miss 2026-06-09).** In the shared maltyweb tree with an active parallel session, a bare `git commit` (or `git add -A`) sweeps the OTHER session's staged work into your commit — proven 2026-06-09: a loss-model commit briefly captured the parallel WeeklyOrders session's staged mig 294 + `import_weeklyorders.php`; recovered via `git reset --soft` + path-scoped recommit, their work intact. **RULE: name the exact files on every commit; `git status` before committing; this is the same discipline as the deploy-pushes-the-working-tree hazard (index BUILD-LAUNCH §line 14) one layer up.** (Reinforces the existing `git add <files> not -A` convention.)
- **COMPUTE-ON-SUBMIT for derivable form fields (NEW 2026-05-28, OPERATOR STANDING RULE, durable).** Any column with a deterministic server-side derivation MUST be computed at form-submit time and written by the SAME insert that persists the row — operator entry is treated as an OPTIONAL OVERRIDE only, never as the trusted primary value. Leaving a derivable column NULL at write-time and relying on a downstream sweep / cron / ingest sidecar to backfill is a CONVENTION VIOLATION. **Why:** (1) silent drift window — form-written rows show NULL in UI/COGS/PPB until the sweep runs (operator sees garbage); (2) multiple write paths to reason about (form + sweep + ingest each touching the same column = the parallel-stores smell, applied to a column); (3) audit ambiguity — `last_modified_by` / revision log can't say cleanly who stamped the final value; (4) **fail-fast on bad formulas** — a wrong server-side formula surfaces on the FIRST operator row a human looks at, instead of propagating silently across N rows like the mig 200 `vendable_hl` incident did. **How to apply:** in the form-submit PHP, AFTER `csrf_verify`+validation+role gate, BEFORE the `bd_*` / event-table upsert, compute the derived value(s); when a canonical view already encodes the formula (e.g. `v_bd_packaging_v2_vendable`), reproduce its formula in PHP (or compute by JOIN against the view's inputs in the same transaction) so the form-write and the view share ONE definition. Operator-typed override path: if the operator entered a non-NULL value, accept it AND stamp the computed value in a sibling `_computed` or audit column so divergence is visible. **Sweep / cron role:** SAFETY NET for non-form write paths (python ingest, bulk loaders, schema additions that retro-add a derivable column, missed-trigger backstop) — NOT the primary path for form rows. The materialized-cache rule (computation-layer-architecture.md §RULE: on-write + scheduled backstop) is the matching pattern one layer down; this rule is its form-input precursor. **Audit at review:** every new form-submit handler PR must answer "which columns are derivable and computed in this handler?" — if the answer is "we'll let the sweep fill it," refuse the merge. **Applies to:** form-packaging.php (vendable_hl — IMMEDIATE retrofit, replaces the deferred sweep), form-brewing.php + form-racking.php + form-fermenting.php + the future session-framework form-submit handlers (audit each for derivable-but-left-NULL columns at build time). **Lifecycle clarification (NEW 2026-05-28, session-framework integration):** compute-on-submit applies PER-SECTION during `op_sessions.phase='in_progress'` (the save handler recomputes the EVENT row + the SESSION-row materialized columns it owns, in ONE transaction — no cascade beyond the session). The SAME formulas re-run as the cascade at `finalize_session($id)` lock-time (the lock-cascade contract, session-model-arc.md §LOCK-CASCADE CONTRACT). Sweep crons are NOT the home for form-derivable columns at any lifecycle phase — neither during `in_progress` (compute-on-submit owns it) nor at `closed` (the lock cascade owns it). The sweep cron's residual role is python-ingest backstop only (catch-all for `session_id_fk IS NULL` rows from legacy Sheets ingest). **RULE-2 reviewer checklist additions for session-framework form-submit handlers (NEW 2026-05-28, durable):** (a) all derivable columns are computed in the form-submit handler (no "we'll let the sweep fill it" tolerated); (b) downstream consumers reading session-grouped events filter by `op_sessions.status='closed'` where they feed a fiscal materialization (or document the exception at the call site); (c) `finalize_session` cascade is idempotent (re-running on an already-closed session is a no-op when recompute output matches stored values); (d) the abandoned-session events policy is honored (operator prompt PURGE-vs-DETACH at abandon-time, recorded in `op_session_steps`; no auto-application; session-less bypass survives §READ-POLICY). These four questions must be answered explicitly in any RULE-2 review of a form-submit handler under the session framework.
- **Gate-helper parity rule (NEW 2026-05-28, durable).** When the SAME gating rule must apply on the input side (compiler/back-end) AND in the UI (operator-facing list/dropdown), the two helpers MUST have identical SQL signatures and stay in sync. Current canonical pair: `_compiler_gated_format_ids()` (in `app/sku-bom-compile.php`, EOF) ↔ `sdc_gated_format_ids()` (in `public/modules/salle-de-controle.php`) — both INNER-JOIN `ref_filler_containers × ref_process_machines × dbc_packaging_format_templates × dbc_container_types`. **Any future gate-rule change MUST update BOTH helpers in the same commit; the comments on each call this out.** Divergent gate helpers are the same smell as parallel data stores: two pieces of code claim to own one rule. PM enforces parity on review.
- **MIGRATION COMPANION SCRIPTS are legitimate tracked infra — NOT the "agents leave helper scripts" anti-pattern (RULED 2026-06-04, mig 263 EPH consolidation).** A migration may ship a same-named `scripts/python/mig<NNN>_*.py` companion when the work needs logic `migrate.php`'s `.sql` `exec()` cannot express — hashing, multi-pass normalization, Python-only libs (pymysql/numpy), anything not expressible as DDL/DML. The companion STAYS TRACKED in the repo as the migration's reproducible record (precedent: `mig263_eph_hash_recompute.py`, the row_hash-recompute half of `263_eph_consolidation.sql`). The "agents leave NO helper scripts" rule forbids EPHEMERAL one-off probes/dispatchers left as litter (those go to `/tmp` + delete) — it does NOT forbid a named, migration-scoped companion encoding irreproducible-in-SQL logic. Distinguishing test: named to its migration + encodes logic SQL can't do + is the audit record of how the data half ran → KEEP. Throwaway probe/dispatcher → /tmp + delete.
- **TWO row_hash SCHEMES COEXIST in `bd_*_v2` tables BY SOURCE (discovered 2026-06-04, mig 263).** BULK tables were ingested by the PYTHON normalizer using **`sha256(json.dumps(canonical, sort_keys=True, default=str))`**; only LIVE-FORM rows (e.g. fermenting id=13433) use the PHP **`bd_row_hash` = `sha256(implode('|', …))`**. So a row_hash on `bd_brewing_*_v2` / `bd_fermenting_v2` / `bd_racking_v2` / `bd_packaging_v2` may have been produced by EITHER scheme depending on Python-ingest vs web-form origin. **Any future migration that recomputes row_hash (e.g. on a recipe_id_fk repoint) MUST detect the scheme per row** — recompute-with-old-key and gate on `== stored` to pick the matching scheme; never blindly apply one. ⚠️ Related pre-existing tech-debt (mig 263 radar): ~21 rows (16 packaging, 5 racking, incl. retro-link + qc-modified lineage) had stored hashes matching NEITHER scheme because `audit_flags` was mutated post-ingest WITHOUT a hash recompute. Standing rule: **any handler that mutates a `bd_*_v2` row's hashed columns after insert must recompute row_hash in lockstep** (same family as the saisie-forms hashCols index-alignment discipline).
