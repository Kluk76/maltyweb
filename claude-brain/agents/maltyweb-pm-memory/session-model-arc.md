## THE CONCEPT (operator, 2026-05-28)
Every input form becomes a 3-phase SESSION: (1) **Début** = STARTUP/QC-FIREWALL — blocking confirm of CIPs + select lots that PASS standards (lagering/CO2/O2…); our eligibility/QC gate made explicit+blocking. (2) **En cours** = ongoing control-data capture (O2/CO2/turbidity/OG/pH/temp) feeding a LIVE session dashboard with real-time flags. (3) **Fin** = end checklist (rinçage, safety CIP, volumes, pertes) + recap. **Multi-operator:** opening a session creates an ACTIVE CARD any operator can resume, showing who opened/validated each step. Per-form variation on one skeleton: fermenting in-progress = OG/pH/temp reads over days; racking loss at END; packaging loss DURING (system SUMS incremental entries).

## 🔑 CARDINAL VERDICT
**A session is a process-instance PARENT that GROUPS events. NOT a state wrapper storing derived facts.** The session row owns ONLY workflow lifecycle (who/when/phase) — no home elsewhere. It stores NO volume/composition/loss/clean-state/cadence-count/measured value — those stay in the event tables, event-sourced/sim-derived. Session = lifecycle envelope; events = truth. Blur this = parallel-store smell at the worst layer.

## DATA MODEL (TWO new tables only — hold the line at two)
**`op_sessions`** — lifecycle envelope (legitimate new table: workflow has no existing home):
- `id BIGINT PK; form_type ENUM('racking','fermenting','brewing','packaging')` (mirrors `bd_cip_events.source_form`); subject (XOR by form_type, house pattern): `vessel_kind ENUM('cct','bbt','yt','tank') NULL, vessel_number INT NULL, recipe_id_fk INT NULL→ref_recipes, batch VARCHAR(32) NULL`; lifecycle: `phase ENUM('start','in_progress','end','closed') DEFAULT 'start', status ENUM('open','closed','abandoned') DEFAULT 'open', opened_by/at, closed_by/at`; house bd_* envelope (`row_hash,is_tombstoned,audit_flags,imported_at,updated_at`).
**`op_session_steps`** — multi-actor audit spine (who/validated each step; append-only):
- `id, session_id_fk→op_sessions, phase ENUM('start','in_progress','end'), step_code VARCHAR(64) (e.g. cip_confirmed/eligibility_passed/volumes_entered), actor VARCHAR(255), acted_at DATETIME(6), payload JSON NULL (confirmation context — passing lots / CIPs confirmed; JSON OK because workflow-audit, NOT a measured fact), row_hash,is_tombstoned,imported_at`.

**Event tables relate via a NULLABLE additive `session_id_fk`** on `bd_racking_v2`/`bd_fermenting_v2`/`bd_packaging_v2`/`bd_cip_events` (+brewing). **Nullable, additive, non-breaking** — every legacy/back-dated/historical row keeps working with session_id_fk=NULL (live counts 2026-05-28: racking 400, fermenting 6675, packaging 2236, cip_events 1093, brewday 803, gravity 8618). Session = OPTIONAL grouping, never a required FK. Same posture as the CIP polymorphic child. Measured values DERIVE from grouped events + TankSimulator filtered by session_id_fk.

**Multi-operator step attribution = `op_session_steps` rows, NOT columns on op_sessions** (a `start_by`/`inprogress_by`… explosion can't represent 2 operators on one phase). Append-only = concurrency-safe by construction.

## STATE MACHINE (start→in_progress→end→closed)
Transitions = `op_session_steps` append + `op_sessions.phase` advance. Writes:
- **open→start:** INSERT op_sessions. **QC firewall (start→in_progress):** CIP confirm → `bd_cip_events` (canonical store, stamped session_id_fk — NEVER a session-CIP table); eligibility = a READ over TankSimulator+qc-thresholds (gates, no new store) + an `op_session_steps` attestation (passing-set in payload). **in_progress capture:** measured rows → EXISTING event tables (`bd_fermenting_v2` Reads / `bd_packaging_readings` / …) stamped session_id_fk; one step row per capture. **end:** losses/volumes → existing `bd_racking_v2` loss cols/safety_cip_done; safety CIP→bd_cip_events. **close:** phase=closed,status=closed; recap DERIVED, stores nothing.

## §READ-POLICY (NEW 2026-05-28 — binary fiscal-read rule on `op_sessions.status`)
**Cardinal rule:** when a downstream consumer reads an event row that carries a NON-NULL `session_id_fk`, it MUST join the parent `op_sessions` and filter on `status`. Three states, BINARY semantics:
- **`status='closed'`** = fiscal-visible. The session is locked, its events are final, downstream materializations (COGS/COP/PPB/Journal-de-bord-closed-feed/sales-COGS/inv_consumption rollups) include them.
- **`status='open'`** = operational-only. The session is still in progress; events are mid-capture; downstream FISCAL consumers MUST EXCLUDE them. Operational live surfaces (the in-cellar tank board, the live Journal de bord feed, the session shell itself) read them through the operational lens, never the fiscal one. (No third `status` value needed for "in-progress vs awaiting-close" — the `phase` axis carries that.)
- **`status='abandoned'`** = invisible-to-fiscal-by-default. The session was opened then abandoned (auto-expired or operator-abandoned). At abandon-time the operator is prompted PURGE-vs-DETACH (see §LOCK-CASCADE CONTRACT below); whichever is chosen, the events do NOT enter any fiscal materialization by default. A subsequent operator-driven re-attach is the only way they re-enter.

**Standing consumer rule (load-bearing):** for every event table carrying `session_id_fk`, any read path that feeds a fiscal/financial/inventory consumer MUST contain either `JOIN op_sessions s ON s.id = e.session_id_fk WHERE s.status='closed'` OR a documented exception (e.g. operational dashboards). The exception MUST be commented at the call site. Two events rows with the same shape but different parent `status` are NOT interchangeable — failing to filter is the same divergence as silently reading mid-write data.

**Consumer + filter audit (load-bearing checklist, audit at every consumer touched):**
- `app/loss-metrics.php` (C7 PPB) → close-filter (fiscal recap).
- `inv_consumption` rollups + `compile_sku_bom_*` (composite/packaging) → close-filter (cost layer).
- `build-cogs.js` / `build-sales-cogs.ts` / `build-month-closure.js` → close-filter (period-frozen).
- `v_mi_cost` / WAC inputs (`inv_deliveries` is session-less, no filter; `inv_consumption` is downstream of session-grouped events → close-filter at the producer, not at the WAC view).
- TankSimulator volume sim → close-filter on session-grouped events; the LIVE tank board surfaces both via a documented dual-read (live-shell reads open AND closed; PPB recap reads closed only).
- `public/modules/sessions.php` Journal de bord PRIMARY view → status-aware: closed → fiscal; open → live; abandoned → hidden by default + toggle.
- `sessions-dashboard.js` view+abandoned toggles → already honors this (atom 6 `da29eb6`).
- `tanks.php` live shells → reads BOTH (operational lens) but flags any volume that includes an open-session contribution as "en-cours" in the UI.

**`session_id_fk IS NULL` events** (legacy, back-dated, historical, python-ingest catch-all) = session-less, BYPASS the read-policy gate — they are TREATED AS pre-framework canonical (fiscal-visible by construction). The whole point of the additive nullable FK is non-breaking compatibility.

## §LOCK-CASCADE CONTRACT (NEW 2026-05-28 — `finalize_session($id)` replaces sweep crons for form-derived columns)
**Cardinal rule:** at the moment `finalize_session($id)` advances `op_sessions.phase` to `closed` + sets `status='closed'`, it ALSO runs the final recompute + CASCADES to every materialized consumer that depends on this session's events. Sweep crons for form-derived columns are RETIRED in the same commit. The cron's residual role is python-ingest backstop ONLY (catch-all for `session_id_fk IS NULL` rows arriving from the legacy Sheets ingest), never the primary path for form-written rows.

**Layer split (load-bearing):**
1. **Per-section save during `phase='in_progress'`** = single-transaction UPDATE of the EVENT row + recompute the SESSION-row materialized columns ONLY (no cascade beyond). This is the COMPUTE-ON-SUBMIT layer (see conventions-and-helpers.md §COMPUTE-ON-SUBMIT). Operator entry is OVERRIDE only; server formula is canonical.
2. **`finalize_session($id)`** at lock = single transaction containing: (a) re-run the per-section recompute over ALL events in the session (idempotent); (b) advance `op_sessions.phase='closed'`+`status='closed'`+`closed_by/at`; (c) CASCADE — kick the downstream recomputes that consume this session's events: SKU-BOM recompile if it sourced any of these events, `inv_consumption` rollup if packaging/brewing rows moved, COGS/COP/PPB invalidation if the session is in a period the materializations have already touched. Atomic: if any step fails, the lock fails + the session stays `open`; idempotent: re-running `finalize_session` on an already-closed session re-runs the cascade harmlessly (writes are no-op when the recompute output matches what is already stored).

**Per-section save vs cascade — discipline (recorded VERBATIM):**
- Per-section save during `in_progress` = compute-on-submit on the EVENT row + refresh SESSION-row materialized columns only. NO cross-session cascade. (Cheap, fast, optimistic; the live shell polls and re-reads it.)
- `finalize_session` cascade = the canonical refresh moment for all downstream materializations. This is the THIRD path beside on-write + scheduled-backstop (see computation-layer-architecture.md §RULE — newly extended).
- ONE-TRANSACTION discipline at BOTH layers. Per-section save = ONE transaction (event UPDATE + session-row recompute + `log_revision`). Lock cascade = ONE transaction (recompute + phase advance + downstream invalidations).
- **Refuse-don't-NULL at the lock boundary on degenerate aggregates** (NEW precedent shape, sibling of compiler-input assertion): if a derived aggregate at lock-time evaluates to NULL/0/undefined where an invariant is violated (e.g. session has zero events, packaged_hl resolves to NULL with no events to sum), `finalize_session` REFUSES the lock + surfaces the specific reason — never silently locks a degenerate session. Caller pattern: lock fails, session stays `open`, an `op_session_steps` row records the reason + the start-firewall card displays it. (Mirrors the BOM line refuse-don't-NULL + the compiler-input assertion in conventions-and-helpers.md §REFUSE-DON'T-NULL.)

**Abandoned-session events policy (operator-prompted, at abandon-time):**
- When a session goes `status='abandoned'` (auto-expire after the SDC-configured N hours, or operator-abandon button), the operator is **PROMPTED** at abandon-time with two choices:
  - **PURGE:** tombstone every event row belonging to this `session_id_fk` (`is_tombstoned=1`) — the events disappear from EVERY downstream read (fiscal + operational + live). For abandonments that represent "false start / wrong session / nothing real happened."
  - **DETACH:** set `session_id_fk=NULL` on every event row belonging to this session — the events survive as session-less / pre-framework-style rows, fiscal-visible by §READ-POLICY's session-less bypass. For abandonments where real events did happen but the session lifecycle envelope itself is being abandoned (operator wants the events kept but unbound).
- The choice is recorded in `op_session_steps` (step_code='abandoned_purge' or 'abandoned_detach' + actor + acted_at).
- Default if the operator dismisses the prompt = NEITHER applied; the abandoned session simply stays invisible-to-fiscal-by-default (§READ-POLICY status='abandoned' lens). PURGE/DETACH are explicit operator decisions, never auto-applied.

**Sweep-cron retirement (the load-bearing scrap):** the sweep cron for `bd_packaging_v2.vendable_hl` (and any future event-derived sweep targeting form-written rows) is RETIRED in the same commit as the form-packaging.php COMPUTE-ON-SUBMIT retrofit. See scrapping-backlog.md entry 20. The cron's RESIDUAL role is python-ingest backstop only — catch-all for `session_id_fk IS NULL` rows arriving from legacy Sheets ingest (those rows are session-less + bypass the lock-cascade by construction). The MATCHING rule one layer down is computation-layer-architecture.md §RULE — newly extended to recognize lock-cascade as the THIRD path.

## §READ-CADENCE (NEW 2026-05-28 — push-vs-poll ruling, RECORDED VERBATIM)

- Fiscal surfaces (PPB / COGS / Journal de bord PRIMARY closed-feed) are period-frozen anyway — auto-refresh would only re-fetch unchanged data; refresh-on-load is correct.
- Promote to push (WebSocket/SSE) ONLY if a future build proves a live surface where 15s lag is operator-unacceptable. Until then = standing scrapping-backlog guard ("don't add push infra"). Record any escalation as an explicit scrapping-backlog entry.

## §WORT CONTRACTS (contrats de moût) — PM RULING 2026-05-28, **CORRECTED 2026-05-28** (BACKLOG, gated)

**1. LEVER — REJECT "category inside famille de levure". USE a BINARY recipe PROCESS-TEMPLATE flag.**

The post-boil cooling target (boil temp → ~10°C) is a continuous value that **varies per contract / per client / per brassin** — one client takes near-boil wort straight to a tanker, another takes it cooled toward ~10°C. PM CALL + WHY:
