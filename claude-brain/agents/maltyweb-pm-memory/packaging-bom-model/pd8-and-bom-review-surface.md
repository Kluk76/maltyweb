## üî¥ THREE SKU IDENTITY LAYERS ‚Äî a packaging EVENT can carry three DIFFERENT SKUs (operator-clarified 2026-05-28, ingest mis-routing bug pending)
Until 2026-05-28 the model treated `ref_skus` as a SINGLE namespace per `bd_packaging_v2` event. **It is NOT.** A single packaging event has up to THREE distinct SKU-identity layers and they can ALL differ on the same row:
1. **Brewing recipe** (`bd_packaging_v2.recipe_id_fk` ‚Üí `ref_recipes.id`) ‚Äî what LIQUID was brewed. Brewing-side aliases re-point `recipe_id_fk` via the ingest's recipe-resolution map (e.g. BLO‚ÜíZEP `ref_recipes.id=57`, BLA‚ÜíMOO `id=44`). Pure liquid identity.
2. **Packaging SKU** (`bd_packaging_v2.sku_id_fk` ‚Üí `ref_skus.id`) ‚Äî what PACKAGE the liquid went into. Production-side output identity. **CAN differ from the recipe-implied SKU** when the recipe is a white-label/relabel: e.g. a ZEP-recipe brew packaged into BLO-branded 4-pack cans ‚Üí recipe ZEP + SKU BLO4C (id 6), NOT ZEP4C (id 56). The packaging-SKU is the canonical handle for everything downstream (COGS, BOM, production dashboards, tank/vendable HL).
3. **Sales SKU** (the code on the customer-facing eshop/B2B sales record) ‚Äî what is SOLD. **CAN differ from the packaging SKU due to UNBOXING** at the eshop layer: eshop opens a ZEPC 24-pack box and re-sells the cans as ZEP4C 4-packs or ZEPCU singles. Lives on the sales feed (`inv_sales_order_lines` / `inv_sales_bc`), NOT on `bd_packaging_v2`.

**Live BLO/BLA white-label examples** (the operator-confirmed semantics):
- **BLO** ("Blonde des Romands") ‚Äî a relabel brand. Recipe 10 (Blonde des Romands as its OWN recipe) was brewed 2021-01 ‚Üí 2022-03 (16 brews only); from **2022-04 onwards BLO is a ZEP-recipe relabel** (brewing alias `BLO ‚Üí ref_recipes.id=57`). BLO packaging SKUs: BLO4=5, BLO4C=6, BLOF=7.
- **BLA** ("Blanche des Romands") ‚Äî ALWAYS a relabel (no own recipe). Brewing alias `BLA ‚Üí MOO ref_recipes.id=44`. BLA packaging SKU: BLA4=4 (inactive). The first live white-label in the BOM model (mig 148; see ¬ßBLA4 WHITE-LABEL).

**Why this matters for ingest ‚Äî the live mis-routing bug.** `scripts/python/ingest_bd_packaging_v2.py` line 333 keys SKU lookup by `(recipe_id, format_suffix)` ONLY. For a BLO event, the brewing-side alias correctly resolves `recipe_id_fk=57` (ZEP), then `sku_map.get((57, "4C"))` returns **ZEP4C (id 56)** instead of **BLO4C (id 6)** ‚Äî the lookup has NO signal for "this is BLO branding." Audited 2026-05-28: **72 events √ó ‚âà1,779 HL of finished product mis-routed across 5 years**:

| Wrong `sku_at_fk` | Raw `neb_beer` text | Events | Total units | ‚âà HL |
|---|---|---:|---:|---:|
| ZEP4C (56) | BLO4C | 28 | 183,656 cans | 918 |
| ZEP4 (55) | BLO4 | 29 | 171,468 bottles | 566 |
| MOO4 (‚âà45) | BLA4 | 6 | 20,616 bottles | 68 |

**Blast radius.** What reads `bd_packaging_v2.sku_id_fk`: (a) sales-driven COGS (`scripts/build-sales-cogs.js` maltytask) ‚Äî beer-level totals shifted between ZEP and BLO; (b) BOM compile ‚Äî already hardened in this session via the gate-helper parity work (commit `de90755`), but its INPUT signal is wrong upstream; (c) production dashboards; (d) any tank simulator or vendable-HL materialization (concurrent agent's mig 200/201 `vendable_hl` work is unrelated to `sku_id_fk`, but downstream consumers of `vendable_hl √ó sku` inherit the bug if not fixed). **Sales side is UNAFFECTED**: `inv_sales_bc` + `inv_sales_order_lines` read sales-channel SKU codes directly with no recipe lookup ‚Äî the divergence is purely production-side mis-attribution.

**üõÝ FIX IN FLIGHT (planned this session, NOT YET BUILT ‚Äî 2026-05-28).** `scripts/python/ingest_bd_packaging_v2.py` ~line 263 (`load_sku_map`) keys by `(recipe_id, format_code)`. Add a **literal `sku_code ‚Üí id` map** queried FIRST; fall back to `(recipe_id, format)` only when raw text is missing/unrecognized:
```python
# Primary: literal raw SKU text ‚Üí ref_skus.sku_code (handles BLO/BLA white-label)

# Fallback: (recipe_id, format) when raw_sku missing/unknown

```
