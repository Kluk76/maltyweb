## CAGE SKU PER-RECIPE COMPLETENESS — every brewed beer needs its own `<X>X` cage SKU (DIBX created 2026-06-18)
> **🔴 INVARIANT (operator-authorized fix, 2026-06-18):** a cage run (`bd_packaging_v2`, format X) on a beer that has NO cage SKU lands with `sku_id_fk=NULL` → `v_bd_packaging_v2_vendable` cannot derive `vendable_hl` (→ **0 HL**) AND the recap/stocktake `stocktake_scope` reads off the NULL SKU → the run falls into the WRONG section ("Bouteille" instead of "Cage"). **Cage SKUs must be COMPLETE per brewed recipe.**
> - **Bug instance:** run `id=6800` (Diversion Blanche, recipe 26 prefix DIB, batch 6, 553 cages) showed 0hl in the packaging recap widget. Root cause was NOT the cage section-split feature — it was a missing SKU: every other beer has its `<X>X` cage SKU (DIVX/recipe25 Diversion, ZEPX, MOOX…) but **Diversion Blanche (recipe 26) had none**. `resolve_packaging_sku_id(recipe=26, format='X')` returned NULL **un-refused** → silent NULL FK on the run.
> - **FIX:** created **`DIBX`** = `ref_skus` id=349 (LIVE-VERIFIED 2026-06-18): `recipe_id=26`, `format_id=6` (the canonical X carrier — `format` VARCHAR shows "Bot" cosmetically, ignore it; format_id is the truth, same as siblings), `units_per_pack=1.0`, `hl_per_unit=0.00330` (per-bottle, post the 2026-06-17 re-denomination above), `stocktake_scope='cage'`, `last_modified_by='web'`, `row_hash` via `compute_row_hash`. Relinked run 6800 → `sku_id_fk=349`. Result: vendable_hl=**1.82 HL**, section=**Cage**. **Insert+relink done in RAW SQL → NO log_revision (ponctual data correction, not a UI write).**
> - 🔑 **REUSABLE RULE:** when a new beer starts being cage-packaged, its `<X>X` SKU must exist BEFORE the run is logged — otherwise the run silently zeroes. The systemic guard (form refuses the unresolved cage submit) is now SHIPPED — see below.

## ✅ form-packaging cage silent-NULL-FK GUARD — SHIPPED + LIVE (option a "refuse à la source", 2026-06-18, maltyweb `db422d3`)
> **The systemic fix for the form-gap behind the DIBX bug above — `form-packaging.php` no longer lets a CAGE run submit with `sku_id_fk=NULL`.** Option (a) "refuse à la source" (PM-preferred, mirrors the refuse-don't-NULL house rule). Previously ANY beer lacking a cage SKU posted a NULL-SKU run → 0 vendable_hl + wrong stocktake/recap section; that hole is now closed for the cage format.
> - **Guard:** inserted in the line-build loop just after `resolve_packaging_sku_id()` — `if ($fSuffix === 'X' && $skuIdFk === null && !(bool)$fIsWl && $fReusesPackagingIdFk === null) { throw new RuntimeException(<message FR>); }`. The throw fires BEFORE any DB write (before `beginTransaction`), caught by the existing `catch (Throwable $e)` → `flash_set('err', pdo_friendly_error($e,'form-packaging'))` + PRG redirect. No 500, no half-written multi-format submission.
> - **Message FR (operator-actionable, per operator request "expliquer le refus pour pouvoir y remédier"):** « Aucun SKU cage (format « X ») n'existe pour « <bière> ». Sans lui, le volume packagé et la base taxe-bière ne peuvent pas être calculés, et le run serait enregistré à 0 HL. Un administrateur doit d'abord créer ce SKU dans Admin → Paramètres → SKU (recette « <bière> », format « X »), puis ce run pourra être saisi. »
> - **Exemptions (deliberate):** white-label (`$fIsWl` — may legitimately lack a Nébuleuse SKU) and cuve-réutilisée (`$fReusesPackagingIdFk !== null` — vendable forced to 0 anyway). Non-cage unresolved rows keep their existing `sku_unresolved` flag unchanged.
> - **Coverage:** create AND edit-mode (same build loop) AND draft/test mode (early refusal is desirable there too).
> - **Review:** fresh-context reviewer passed — 0 BLOCKER; 1 WARNING (double-escape, `flash_render()` already HTML-escapes) → fixed before commit.
> - 🔴 **KNOWN LIMIT (noted, not a bug):** the guard covers ONLY the cage format `'X'`. Other unresolved formats stay on the `sku_unresolved` flag (triage, not refusal) — deliberate (tight blast radius; ONLY cage silently under-reports volume + fisc, because its NULL SKU also wrong-sections the run). Extending the hard refuse to other formats is a SEPARATE decision if ever wanted.

## 🔴 CRUCIAL — MULTIPACK → `-X` CAGE STOCK-PULL, BY SALES CHANNEL (COGS/COP routing rule; operator-flagged crucial 2026-05-26)
Multipacks (4PB / 4PC / 6×4 …) are sourced **differently by fulfilment**, and COP/COGS MUST reflect it — **the bottle+cap+label cost lives ONCE, inside `-X`** (do NOT double-count it, do NOT model unbox-waste). The discriminator is two-pronged:
- **UNBOXED (built from loose individual units, NOT drawn from `-X`)** — exactly two cases:
  - **BC client ACCOUNT 1080 (privé) sales**, and
  - **Shopify orders PICKED UP at the brewery** (pickup, as opposed to shipped).
- **PULLED FROM `-X` cage stock — EVERYTHING ELSE:** Shopify **shipped** orders **+ all other clients/channels** (the bottle+cap+label cost already booked in `-X`; the multipack draws it DOWN).
- **Data discriminator for the build:** `account = 1080` **OR** `Shopify fulfilment = pickup-at-brewery` → unboxed; otherwise → `-X`. ⚠️ This means the **Shopify sales feed must carry a pickup-vs-shipped flag** — without it the routing can't be computed. Privé is already identifiable (account 1080).
- **Implementation = part of the FUTURE multipack-stock-pull build** (Stage-2-adjacent — sales/eshop consumption matrix, derivation-tree-and-schema.md Stage-2). Recorded NOW so the channel-routing rule isn't lost. The cost-lives-once-in-`-X` invariant is the cardinal correctness constraint: a `-X`-pulled multipack must NOT re-add bottle/cap/label cost (it's a draw-down from `-X`), while an unboxed sale (acct-1080 OR Shopify-pickup) consumes loose units directly. Cross-link: this is the consumption-side mirror of the `-X` WIP item above.
- **Cross-link to the THIRD decomposition axis (sku-decomposition-tree.md):** the per-recipe HL-share decomposition (`ref_sku_composite_slots`) computes a pack's THEORETICAL per-recipe ingredient map; THIS stock-pull rule is the ACTUAL-consumption routing that says which physical stock that map draws from. A `-X`-pulled multipack's per-recipe liquid attribution is a DRAW-DOWN from that beer's `-X` (already-booked), not a fresh consumption — the per-recipe attribution must not double-book it. A MIXED deco pack spans multiple beers → it can only draw from multiple per-beer `-X` cages OR be built from loose units, never from one `-X`.
