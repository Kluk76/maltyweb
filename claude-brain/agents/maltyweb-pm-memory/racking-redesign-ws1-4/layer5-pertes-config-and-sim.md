### LAYER-5 PM RULINGS

**R1 — CONFIG MODEL (commissioning_settings, section='pertes').** Key/value, NO ref-table (these are scalar policy constants + bands, exactly the qc_thresholds class). 7 keys, ALL with default_num = current behavior (ZERO drift):
- `pertes_racking_loss_hl` value_num=0.9000 default_num=0.9000 unit_fr='HL' — the fixed racking process loss (de-hardcodes RACKING_LOSS_HL).
- `pertes_packaging_loss_hl` value_num=0.1500 default_num=0.1500 unit_fr='HL' — the fixed packaging process loss (de-hardcodes PACKAGING_LOSS_HL).
- `pertes_rack_warn_pct` value_num=… unit_fr='%' — rack-stage loss-% palier (operator sets; seed a sane default e.g. 5).
- `pertes_packaging_warn_pct` unit_fr='%' — packaging-stage palier.
- `pertes_brewing_warn_pct` unit_fr='%' — brewing-stage palier.
- `pertes_total_effectif_warn_pct` unit_fr='%' — total vs cast-out palier (the 18% example bucket).
- `pertes_total_nominal_warn_pct` unit_fr='%' — total vs nominal palier (the 10% example bucket).
GLOBAL per-stage (per-recipe = later phase, mirroring QC-thresholds). Thresholds are WARN-only (one palier each — operator chose %, single band, not warn+outlier two-band like QC). is_active=1. Seed via the de-hardcode migration (mig 184) so the keys exist before any UI.

**R2 — DE-HARDCODE THE SIM (the COGS-critical one).** Replace the two `private const` with values read ONCE at TankSimulator CONSTRUCTION (not per-event — no per-event DB hits). Add `private float $rackingLossHl; private float $packagingLossHl;` set in __construct via a single `commissioning_settings WHERE section='pertes' AND key_name IN(...)` SELECT, COALESCE to the 0.9/0.15 defaults if a key is missing/inactive (same defensive-default pattern as qc_global_bands). Replace `self::RACKING_LOSS_HL`→`$this->rackingLossHl` (L378) and `self::PACKAGING_LOSS_HL`→`$this->packagingLossHl` (L492, L510). **VERIFICATION GATE (hard): with the seeded defaults (0.9/0.15) the sim output must be BYTE-IDENTICAL to pre-change** — diff a full sim run (all BBT/CCT states) before vs after; this path feeds COGS/WIP and MUST NOT drift. The constructor read is the ONLY DB hit; the values are instance state thereafter.

**R3 — TOTAL-LOSS = DERIVED (view/resolver), event-sourced, NOT stored.** Numerator = packaged (FG) volume for the batch. Build it NOW as a resolver/view (operator wants it tracked), reading the event stream — no new column. Data sources + join keys:
- packaged_hl = SUM over packaging events for the batch. JOIN key = `(beer/recipe, batch)`. Packaging HL source = bd_packaging (the sim's PACKAGING `$e['hl']`); the canonical packaged volume per batch. (NB bd_packaging join keys: recipe_name canonical, beer/neb_beer hold SKU codes — see reference_bd_packaging_join_keys.)
- cast_out_hl = `bd_brewing_gravity_v2.final_volume WHERE event_type='Cooling'` summed per `(beer, batch)`.
- nominal_hl = `ref_brewhouse_size.size_hl` effective at the brew/cooling date (× n_brews if a batch is multi-brew — match the cast_out aggregation grain). 
- **loss_vs_effectif = (cast_out_hl − packaged_hl) / cast_out_hl**; **loss_vs_nominal = (nominal_hl − packaged_hl) / nominal_hl**. Both ÷0-guarded.
- Join chain: brewday (beer,batch,cct,event_date) ↔ gravity Cooling (beer,batch,final_volume) ↔ racking (neb/contract beer+batch) ↔ packaging (recipe_name,batch). Batch is the spine. Shape = a `v_batch_loss` view (or a resolver `app/loss-metrics.php` if the joins need normalization the sim already does). Prefer a VIEW if the joins are clean SQL; resolver if it needs the sim's beer-name normalization (likely — Div.Blanche vs Diversion Blanche). LEAN to a resolver that reuses TankSimulator's keying, exposing per-batch {cast_out, nominal, packaged, loss_vs_effectif, loss_vs_nominal}.

**R4 — PER-STAGE FORMULAS (the crux, nailed):**
- **Rack stage** = (explicit operator losses) / (volume that left the source). Numerator = `loss_source_hl + loss_dest_hl` (the WS1 operator-entered cols). Denominator = `racked_vol_hl` (the volume transferred). **DOES NOT include the fixed RACKING_LOSS_HL** — that 0.9 is a baked-in process constant (already in netRacked), not an exceptional event; the rack-stage flag measures EXCEPTIONAL operator-recorded loss as a % of the transfer. Formula: `(loss_source_hl + loss_dest_hl) / NULLIF(racked_vol_hl,0)`. Flag if > pertes_rack_warn_pct.
- **Packaging stage** = (fixed packaging loss + any explicit packaging loss) / packaged volume. Today there's only the fixed PACKAGING_LOSS_HL and no explicit packaging-loss input. Formula now: `packaging_loss_hl / NULLIF(packaged_hl,0)` where packaging_loss_hl = the fixed constant (× n packaging events) + future explicit packaging-loss col. Flag if > pertes_packaging_warn_pct. (When/if an explicit packaging-loss input is added, it adds to the numerator — leave the slot.)
- **Brewing stage** = see R5 (needs a new input; until then the flag is dormant/derivable only as nominal−cast_out).
- **Total** = R3's two variants vs pertes_total_effectif_warn_pct / pertes_total_nominal_warn_pct.

**R5 — BREWING-STAGE LOSS = DERIVED, not a new operator input (minimal addition).** The brewery already has the two volumes to derive brewing loss WITHOUT a new form field: **brewing loss = (nominal_hl − cast_out_hl) / nominal_hl** = `(ref_brewhouse_size.size_hl − Cooling.final_volume) / ref_brewhouse_size.size_hl`. This is the gap between what the kit was sized/dosed for and what actually made it into the CCT cold (evaporation, trub, dead-space, transfer-to-CCT loss). NO new column, NO brewing-form input — both operands already live in the schema. **This makes the "vs-nominal total" and the "brewing stage" share the nominal−cast_out term — by design** (brewing loss is the nominal→cast_out leg; rack+packaging are the cast_out→packaged legs; they compose into the vs-nominal total). Flag if brewing loss > pertes_brewing_warn_pct. ⇒ "wire it on the brewing form" = surface the DERIVED brewing-loss % + flag on the brewing form's submit dialog (FormFramework warning), reading nominal (ref_brewhouse_size) vs the cooling volume the operator just entered — NOT a new captured loss field. IF the operator later wants an EXPLICIT brewing-loss input (e.g. measured spillage distinct from evaporation), that's one new nullable col on the cooling/gravity row — reserve, don't build. **Schema delta = ZERO for the derived version.** (Open Q for operator: is the cooling final_volume entered on a form FormFramework controls today? It is NOT on form-brewing.php — it's a separate gravity path. So the brewing-stage FLAG may have to live on whatever form captures Cooling final_volume, or on the KPI surface only, until the gravity capture is a FormFramework form. Flag this — see DIVERGENCE.)

**R6 — FORCED-COMMENT INTEGRATION (cross-form, the loss-flag → loss_note route).**
- The loss-threshold palier is an **outlier-class warning** in FormFramework terms (it forces a mandatory comment). BUT its comment target is `loss_note`, NOT the generic `fw_comment`. RULING: extend FormFramework so a warning can declare a `commentTarget` (default `fw_comment`); the loss-flag warning sets `commentTarget:'loss_note'`. On confirm, if the dialog comment is non-empty AND loss_note is empty, inject the comment into `input[name="loss_note"]`/the loss_note textarea (mirror the existing fw_comment injection at L283-290, but target-aware). If loss_note already has content, don't clobber — the gate is "WARNING fired AND loss_note empty → force". This keeps the loss narrative in the canonical loss_note column (queryable beside loss_cause for KPI), not buried in a generic comment.
- **Minimal-touch alternative if extending FormFramework is too invasive:** the racking form's own submit handler reads the dialog comment and, when a pertes-palier warning fired and loss_note is empty, writes it to loss_note before POST. PREFER the FormFramework `commentTarget` extension — it's the reusable cross-form mechanism (brewing/packaging will need the same), matches the "CIP module = reusable pattern" philosophy, and avoids per-form copy-paste.
- **Keep the DISTINCT hard "impossible" check (already ruled WS1):** loss > available volume = a hard DATA-ERROR (loss_source > source CCT vol, loss_dest > dest residual). This is SEPARATE from the soft palier — it's a different warning class. WS1 ruled it soft-flag-not-block (QA-outlier posture, import-then-flag). So BOTH exist: (a) soft palier-% flag → forces loss_note comment; (b) soft over-volume "impossible" flag → audit_flags + UI warn. Neither blocks submit. The palier one additionally forces the comment; the impossible one is its own warning line. Do NOT conflate.

### LAYER-5 DIVERGENCE FLAGS
