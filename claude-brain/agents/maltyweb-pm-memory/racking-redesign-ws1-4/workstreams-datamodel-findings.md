## THE 4 WORKSTREAMS — PM RULINGS

### WS1 — Losses = SEPARATE "Pertes" section (decoupled from interrupted)
**Semantics (operator-confirmed):** loss_source = destroyed at source (reduces source, never arrives); loss_dest = destroyed at/in destination (reduces dest); ADDITIVE beside racked_vol, NOT a slice of it. Post-event: source = V_source − racked_vol − loss_source; dest = residual + netRacked − loss_dest.
- **"Transfert interrompu" toggle = reveals a Reason textbox ONLY** (reason required when interrupted=1). NO loss flags here. (This is the old ruling's reason field, minus the dropped beer_lost.)
- **NEW columns on bd_racking_v2 (mig 183):** `interrupted_flag TINYINT(1) NOT NULL DEFAULT 0`, `interrupted_reason VARCHAR(255) NULL`, `loss_source_hl DECIMAL(8,3) NULL`, `loss_dest_hl DECIMAL(8,3) NULL`, `loss_cause ENUM('produit','machine','humain') NULL`, `loss_note VARCHAR(255) NULL`. (All raw EVENT INPUTS — operator-stated facts, not derived. Refuse-don't-NULL: NULL = "no loss recorded", a real semantic absence, NOT an unresolved link.)
- **RULING — ONE loss_cause/loss_note pair for the section, NOT one per side.** Operator wants COARSE categorization for KPI. A loss event on one transfer is one operational incident with one root cause (a product issue, a machine issue, a human error) even if it manifests at both source and dest. Two separate cause pickers = false precision + double the UI for a coarse KPI. If a future need to attribute cause per-side appears, that's a deliberate extension (it'd argue for a child `bd_loss_events` table) — do NOT pre-build it. ONE pair. The two volume fields (source/dest) stay separate (they're distinct physical measurements).
- **RULING — {Produit/Machine/Humain} = FIXED ENUM, not a ref table.** Operator explicitly wanted COARSE 3-bucket categorization (NOT a fine taxonomy, NOT free-text-only). Three buckets that are conceptually closed (the universe of loss root-causes at this grain) + change on the order of years = a fixed ENUM, mirroring the `bd_cip_events.target_code` machine-list precedent (operator named a closed set, didn't ask for editability → ENUM, a 4th = one-line ALTER). This does NOT violate "never hardcode dropdowns": that rule targets OPERATIONAL master-data lists the operator maintains (CIP types, suppliers, recipes). A 3-value closed analytical category that the operator deliberately fixed is an ENUM domain, same class as `racking_destination_type`/`loss_cause`. The free-text `loss_note` carries the detail. If the operator later asks to manage the list → promote to `ref_loss_causes` (cheap). Default: ENUM.
- **Soft over-volume validation, NOT block** (operator-confirmed): loss_source ≤ available source volume, loss_dest ≤ available dest volume → flag in `audit_flags` + UI warn, never reject. Available volumes read from TankSimulator (source CCT `volume_hl`, dest residual = blend_hl). This is the QA-outlier posture: import-then-flag.
- **Live volume-balance preview** = extend `updateResultant()` → show source-after / dest-after using the loss inputs (pure JS display, nothing stored). The canonical post-event volumes are derived by the SIM, never stored (A1 discipline generalizes).

### WS2 — CIP on interrupted transfer (per-BBT clean/dirty state)
- interrupted=1 AND racked_vol=0 (nothing moved) → reveal **"BBT encore propre ? Oui/Non"**. Oui → BBT stays clean in the system. Non → BBT dirty → next fill requires CIP.
- interrupted=1 AND racked_vol>0 → BBT received volume → normal volume + normal CIP logic (no clean-flag question).
- **RULING — EVENT-SOURCED, no stored per-BBT state column.** A "current clean/dirty" column on ref_bbt = derived-current-state = the parallel-store smell (exactly the no-stored-current-volume rule, applied to clean-state). The clean/dirty status of a BBT at any time is DERIVABLE from the event stream: the latest of {a destination-vessel CIP event for that BBT in bd_cip_events} vs {the latest fill/rack-into that BBT}, PLUS the new "BBT propre? Oui" signal on an interrupted-zero-rack row. 
- **NEW column (mig 183): `dest_bbt_still_clean TINYINT(1) NULL`** on bd_racking_v2 — a raw EVENT INPUT ("operator attests the BBT was untouched by this aborted rack"), captured ONLY on the interrupted+racked_vol=0 branch, NULL otherwise. NOT a derived current-state column. The next transfer's clean-state DERIVATION reads: was there a vessel-CIP event on this BBT since its last real fill? OR was the last touch an interrupted-zero-rack with `dest_bbt_still_clean=1`? → clean. This is one event input feeding a derivation, not a mutable status flag.
- **Derivation home:** a small resolver (extend TankSimulator OR a sibling `app/bbt-clean-state.php`) that, given a BBT number + as-of date, returns clean|dirty from (bd_cip_events vessel-CIP for that BBT) + (bd_racking_v2 fills + interrupted-clean attestations). The next transfer's CIP gate reads it. Keep event-sourced. **Machine CIP always-required (unchanged) — this rule is BBT-vessel-only.**
- Interplay with A2 (blend rule, already live): A2 says residual>0 ⇒ dest CIP optional. WS2 adds: an interrupted-zero-rack that left the BBT clean ⇒ next fill's dest CIP optional too. Both are "the vessel is already in a usable state" exceptions to dest-CIP-required. Compose them in the dest-CIP-required predicate, don't fork it.

### WS3 — Destination auto-blend + multi-lot composition (the deep one)
- Dropdown BBT/CCT/YT + tank number STAYS. Empty BBT or DIFFERENT beer = ONLY via hors-process.
- Once BEER selected → form auto-surfaces **non-empty BBTs already holding the SAME beer** (blend candidates) with current contents → REPLACES the manual `blend_hl` typing. Operator picks a candidate BBT; its current residual auto-fills blend_hl.
- **RULING — residual READ from TankSimulator AND persisted to blend_hl = LEGITIMATE EVENT-INPUT CAPTURE, confirmed.** This does NOT violate "never store derived current volume." The distinction: `blend_hl` is a SNAPSHOT of the residual AT TRANSFER TIME — an immutable historical fact about that event (what was in the tank when this rack happened), captured into the event row. It is read FROM the sim (which is the live authority) at capture time, then frozen. It is NOT a mutable "current BBT volume" column that anything updates as the tank drains — nothing ever UPDATEs blend_hl after the row is written. That's the same pattern as capturing a meter reading: the live value is authoritative, the captured value is a historical event input. The sim KEEPS deriving live volumes from the event stream (including this row's racked_vol+blend); blend_hl never becomes a competing live-volume source. **Approved: read-from-sim, persist-as-event-snapshot, never mutate.**
- **RULING — multi-lot composition: the sim is ALREADY multi-lot (`blend_info`); the build is to (a) make it lot-faithful on drawdown, (b) make composition queryable, NOT to add a parallel store.** Today `blend_info` rescales on blend-in but PACKAGING drawdown only touches `volume_hl` (leaves blend_info stale). Foundational fix: PACKAGING + loss_dest must proportionally decrement each lot in `blend_info` (draw down all lots pro-rata by their current share). Then per-lot % = `lot.vol / sum(blend_info.vol)` is derivable at any sim instant.
  - **Composition is EVENT-SOURCED, NO new persisted composition table.** The event stream (COOLING fills, RACKING blend-ins with blend_hl snapshots, PACKAGING drawdowns, loss events) IS the source of truth; the sim reconstructs composition by replay. A persisted `bbt_composition` snapshot table = the parallel-store smell (a derived-current-state store that drifts from the events). The directional asks (live per-lot % in a blended BBT; blend ratios of packaged blended lots) are BOTH reconstructable from the replayed `blend_info` — they need a sim that exposes its `blend_info` (a read API / a computed view materialized on-demand), not a new write store.
  - **PACKAGING blend-ratio attribution (directional, NOT built now):** when a packaged lot draws from a blended BBT, its ingredient/COGS attribution should split across the constituent lots by their `blend_info` shares at draw time. This is the eventual coupling into the SKU/BOM derivation tree (per-recipe HL share = the THIRD decomposition axis, already flagged in sku-decomposition-tree.md). Model must NOT preclude it → keeping composition event-sourced + lot-faithful in the sim is exactly what enables it later. Do NOT wire COGS attribution now; just don't break the path.
- **How events update composition (the model shape):** RACKING blend-in = append lot + rescale (exists); PACKAGING draw-down = pro-rata decrement all lots (FIX); loss_dest = pro-rata decrement all lots (NEW, same mechanic as packaging draw); loss_source = decrement source CCT, never reaches BBT (NEW). Below BBT_EMPTY_THRESHOLD → BBT cleared (exists). This is all in-sim; no schema change for composition itself — only the WS1 loss INPUT columns feed it.

### WS4 — BBT blend CIP cadence (6→acid→6→full→cycle)
- Configurable in Salle de Contrôle: 6 racks without BBT-CIP → acid CIP; then 6 more → full CIP; then cycle repeats.
- Counter increments on **ANY rack into a BBT where "CIP BBT effectué" is NOT checked** (with or without residual — they rack into empty non-CIP'd BBT if same yeast strain). Acid CIP resets the acid counter; full CIP resets everything.
- **RULING (a) — per-BBT counter = EVENT-SOURCED from bd_cip_events + bd_racking_v2, NOT a stored counter.** A stored `racks_since_cip` int on ref_bbt = mutable derived-current-state = parallel-store smell (and drifts the instant a historical rack is back-dated or tombstoned). Derive: for a given BBT, count racks-into-it (bd_racking_v2 dest=BBT for that number) since the last vessel-CIP event of the relevant type in bd_cip_events. Two counters, both derived: racks-since-last-acid-or-fuller-reset, racks-since-last-full. Back-dating + tombstoning stay correct for free (the derivation re-reads the stream).
- **RULING (b) — acid-vs-full reset mapping = config, lives in commissioning_settings (NOT hardcoded).** Which ref_cip_types count as an "acid reset" vs a "full reset" is operator-configurable (the seed has Soude/Acide/Full CIP/Full CIP + rinser; the mapping of those onto acid-reset/full-reset is policy). Store as a value_text list of cip_type ids/names per reset class. Full CIP (and Full CIP + rinser) → full reset (resets both counters); Acide → acid reset (resets acid counter only). Soude (caustic) → operator decides its reset class. The cadence resolver reads this mapping + the thresholds from commissioning_settings; nothing hardcoded.
- **RULING (c) — Q4c: thresholds + acid/full mapping live in the EXISTING `#sec-cip` section of Salle de Contrôle, NOT a new "Transferts" section.** Justification: `#sec-cip` is ALREADY the CIP master-data home (ref_cip_types list-CRUD lives there), and the cadence policy IS CIP master-data — it governs CIP scheduling, references ref_cip_types directly (the acid/full mapping), and reads/writes commissioning_settings (the same pattern qc_thresholds uses). A separate "Transferts" section would (i) split one coherent CIP-policy surface across two homes, (ii) imply the cadence is racking-specific when it's a BBT-vessel-CIP policy that any BBT-filling form should honor. Add the cadence config as a second panel WITHIN `#sec-cip` (thresholds + the reset-class mapping picker over ref_cip_types). NEW commissioning_settings keys, section='cip_cadence': `cip_cadence_acid_after` (value_num=6), `cip_cadence_full_after` (value_num=6), `cip_cadence_acid_reset_types` (value_text=csv of cip_type ids), `cip_cadence_full_reset_types` (value_text=csv).
- **NEVER blocks past threshold → SOFT FLAG** at three surfaces (operator-confirmed): (1) blend-candidate selection in the form (the candidate card shows "N racks since CIP — acid CIP recommandé"), (2) submit (audit_flags warn, never reject), (3) KPIs (the racking KPI dashboard). All read the same event-sourced cadence resolver.
