### VERDICT (the 5 answers)
1. **Forms in scope (PHP + fields):**
   - **`public/modules/form-packaging.php` = PRIMARY** (the literal conflict site). Cuv litre fields `formats[N][loss_keg_liquid_l]` + `taproom_keg_l` (litres — an HL value reads ~100× low). The per-unit count fields `formats[N][prod_total_units]` / `qte_unites` feed `compute_packaging_vendable_hl` via `hl_per_unit/units_per_pack` → a magnitude slip there silently mis-scales vendable_hl/beer-tax. run_type ∈ {bot,can,can33,keg,cuv}; each row carries `run_type` + `format_suffix` → the selected `ref_packaging_formats` row is known client-side.
   - **`public/modules/form-brewing.php`** — `cool_final_volume[]` (HL). Expected band = CCT capacity (30–240 HL live) / brewhouse size 30 HL; a litres-as-HL or HL-as-litres slip is catchable.
   - **`public/modules/form-racking.php`** (handler `public/api/racking-phase-submit.php`) — `racked_vol_hl`, `blend_hl`, `flowmeter_start_hl`/`flowmeter_end_hl`, `loss_source_hl`/`loss_dest_hl` (all HL). Expected band = serving-tank 5/10/30 HL + CCT/BBT 30–240 HL.
   - Fermenting carries no headline volume field → out of scope for v1 (readings are gravity/pH/temp, already threshold-guarded).
2. **Canonical magnitude source — `ref_packaging_formats(run_type, hl_per_unit)`.** This table ALREADY encodes the expected per-unit volume per format (cuv V id18=0.01→1 L/unit; keg F id15=0.20→20 L; bot BU=0.0033→0.33 L; etc.). **No new table, no hardcode.** Per-unit expected litres = `hl_per_unit × 100`. For the HL-denominated brewing/racking fields, the vessel-capacity tables are the band: `ref_cct`/`ref_bbt` (30–240 HL), `ref_serving_tanks.capacity_hl` (5/10/30), `ref_brewhouse_size.size_hl` (30). A "expected magnitude per format" need NOT be invented — derive the order-of-magnitude window from these canonical numbers (e.g. ±1 order of magnitude around the format's nominal). Container atoms live in `dbc_container_types.volume_l` if a finer per-container floor is wanted.
3. **Soft confirm-at-submit, NOT hard block** — operator's lean is correct and matches house philosophy (import-then-flag, never reject) AND the real partial-liner-fill case (we amortize on announced volume, not nominal capacity → a genuinely-low cuv fill must be enterable). Confirm-with-reason override, exactly like the existing `co2o2InversionConfirmed` window.confirm + bypass-flag and the `edit_shared_tank_confirmed`/`tank_reading_override` server tokens.
4. **Architecture — it is NOT a new mechanism; it's populating an empty `thresholds:{}`.** All 4 saisie forms already wire `FormFramework.init()` (`public/js/form-framework.js`) which provides per-field `warn`/`outlier` ranges (`evaluateField`), a `detectSwaps` order-pair detector, `extraWarnings()` hook, and a confirm-with-comment dialog that **requires a comment when any warning is level 'outlier'** (the soft-confirm we want). Brewing & packaging pass `thresholds:{}` (empty) today; packaging/racking also already use the `extraWarnings` + comment-routing path. **BUILD = (a) a shared magnitude validator** that, per the selected format/run_type, looks up the format's nominal unit-volume and flags an entered value that is off by ~1 order of magnitude (the "seems hl instead of l" case) as an `outlier`-level warning with the operator-facing message — wire it through `extraWarnings()` (per-format, because the format is a runtime selection, not a static field threshold) **+ (b) a server-side soft-confirm token** mirroring the existing `*_confirmed=1` pattern so the override is auditable (and a forged/garbage bypass can't poison a value). The per-format nominal must reach the client: either inject a `window.FORMAT_MAGNITUDES` map (id/run_type → hl_per_unit) from PHP (sourced from `ref_packaging_formats`), or read it off the already-present `data-*`/mosaic dataset the packaging form carries per format. Server reads the same `ref_packaging_formats` row to re-evaluate (never trust the client's claim of "in range"). **Roadmap fit:** independent of recette v2 (#0c DONE) and of the -V compiler revert (#0 COGS cutover), BUT **sequence AFTER the -V revert lands** so the canonical per-litre basis it validates against is already live — building it before would risk codifying the very ambiguity (per-HL vs per-litre) the revert resolves. No conflict with anything in flight; parallelizable with WeeklyOrders ingestion (#0a4). Honors: CSS/JS in /public, no inline; RULE-2 review before land; self-cleaning synthetic round-trip; COGS-invariant not needed (no qty-math change — this only ADDS a pre-submit nudge, the stored value on confirm is byte-identical to today).
5. **EQUIP: ui + coder + sql** (sql = read the per-format thresholds from `ref_packaging_formats`/vessel tables; ui = the FormFramework threshold/extraWarnings wiring + dialog; coder = the server soft-confirm token) + **webapp-testing** smoke step (submit a deliberately-100×-off cuv value, assert the confirm fires; submit a genuine partial fill + override, assert it writes).

### FOOTGUNS to brief the agent on
- The packaging cuv path: `prod_total_units`/`qte_unites` are INDIVIDUAL-ITEM counts for bot/can/keg but cuv V is **LITRES** (volume-dimension.md §vendable_hl). So the cuv magnitude check is on a LITRE quantity, not a unit count — the threshold logic must branch on run_type='cuv' (litres) vs item-count formats.
- Do NOT make the band so tight it false-flags legit partial liner fills or a small pilot brew — the check is ORDER-OF-MAGNITUDE ("~100× off"), not a precision range. A 30% under-fill must pass clean; a 100× slip must flag.
- Server re-evaluation must read `ref_packaging_formats` itself (canonical), not trust a client-passed nominal — same discipline as the tank-reading override (server enforces, client only requests).
- Edit-mode: a correction re-opening a past row must NOT re-fire the magnitude confirm on values that already passed/were-overridden once (or it nags on every edit). Carry the override token / suppress on unchanged values.

## §SAISIE BRASSAGE 1055 HOTFIX — form-brewing POST-side recipe resolver (✅ SHIPPED + LIVE 2026-06-16)
**Bug:** regression `9dc0a8a` (deployed 2026-06-15) had rewritten the POST-side server recipe resolver in `public/modules/form-brewing.php` (~L80-93) to `SELECT id FROM ref_recipes WHERE name=? AND is_active=1 GROUP BY name HAVING COUNT(*)=1` — non-aggregated `id` under a `GROUP BY name` violates MySQL 8 `ONLY_FULL_GROUP_BY` → **error 1055 on every POST**.
**Severity = HARD-BLOCK, not edge case:** Gonzalo's data-entry pattern is to create a brewday header then repeatedly re-open via "Corriger/modifier" and re-submit to add datapoints incrementally. The resolver runs UNCONDITIONALLY on EVERY POST, so once `9dc0a8a` deployed, every `corriger` on his in-progress brewday threw 1055 → hard-blocked mid-entry. **This workflow is now a STANDALONE STANDING FACT at the top of this file (§STANDING OPERATIONAL FACT — operator incremental-entry workflow) — reusable across ALL form work, not specific to this bug.**
**Fix (option = minimal deterministic rewrite — PM-recommended; applied by Kouros directly, NOT an agent):** plain `SELECT id FROM ref_recipes WHERE name=? AND is_active=1` → `fetchAll()` + `if (count($rows) !== 1) throw` (same French "introuvable ou ambiguë" message) + `$recipeId = (int)$rows[0]['id']`. **NO `ANY_VALUE`** — the "refuse on ambiguity / never guess" contract is preserved EXACTLY; the resolved `recipe_id_fk` stays deterministic (it's the load-bearing FK feeding every downstream tank/consumption/COGS join). Re-runs cleanly on each `corriger` → Gonzalo's incremental-edit loop works again.
**Deploy:** surgical single-file rsync of `form-brewing.php` ONLY (NOT `bin/deploy.sh --apply`) because the shared tree had unrelated WIP (kpi-handlers.php, expeditions.css, mon-tableau.css, kpi-charts.js) that must not go live — the codified dirty-shared-tree workaround.
