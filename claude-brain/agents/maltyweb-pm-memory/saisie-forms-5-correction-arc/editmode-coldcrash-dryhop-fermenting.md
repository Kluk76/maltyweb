## §SAISIE EDIT-MODE CORRECTION PATTERN — `?edit=<id>` (proven cross-form: brewing + packaging + fermenting + RACKING — ALL 4 forms shipped 2026-06-04, working tree uncommitted)
The reusable "Corriger une saisie" arc — re-open a past `bd_*_v2` row, prefill the form, re-POST through the SAME idempotent handler (NO new endpoint). Now proven on ALL FOUR saisie forms (racking closed the last gap 2026-06-04). Canonical mechanics (honor on every future form's edit add):
- **Entry:** per-row "Corriger" link/column on the recent-submissions surface → `?edit=<row id>`. For brewing/packaging/fermenting this is the `*-phase-recent.php` partial; for racking it is the form's OWN inline recent table (~L1815) since the monolithic form has no recent partial. The module page (`form-<x>.php`) does GET detection `$editId`/`$editMode`, loads the row by PK with `is_tombstoned=0`, builds `$prefillEdit` + `$editBanner`, and **captures the original `submitted_at`** into a hidden field (`edit_submitted_at`) + hidden `edit_id`.
- **THE LOAD-BEARING INVARIANT — preserve `submitted_at`:** the handler sets `$submittedAt = $isEditMode ? $editSubmittedAt : date('Y-m-d H:i:s.u')`. Original `submitted_at` is PART OF THE NATURAL KEY → preserving it makes `bd_upsert` UPDATE-in-place. A fresh `date()` in edit mode = the bug: it changes the NK → INSERTs a DUPLICATE instead of correcting. **Always regex-validate `edit_submitted_at` (strict `Y-m-d H:i:s.u` shape) before trusting it** — it's a hidden field, an attacker/garbage value must not poison the NK.
- **⚠️ NK-FIELD EDIT ≠ in-place correction under `bd_upsert` (RULED 2026-06-04, packaging `?edit=6737` ZEP210→211).** Preserving `submitted_at` only guarantees UPDATE-in-place when the OTHER NK cols are unchanged. `bd_upsert` is `INSERT … ON DUPLICATE KEY UPDATE` keyed on `uq_natural_key` AND `uq_row_hash`. Editing ANY NK field — `neb_batch`, `neb_beer`, `nebuleuse_format_suffix` (pkg), `seq` (racking) — yields a NK that collides with NEITHER unique key → MySQL **INSERTs a new row, leaves the original orphaned = DUPLICATE, not a correction.** (`bd_packaging_v2` NK = submitted_at,neb_beer,neb_batch,contract_beer,contract_batch,row_origin,nebuleuse_format_suffix; `bd_racking_v2` NK = …,seq. Both include neb_batch.) All 4 forms share this latent flaw; today they're safe ONLY because identity is card/hidden-field-driven and the operator can't freely retype it. **To actually correct an NK field: UPDATE BY PK (`edit_id`), NOT upsert-by-NK** — `UPDATE bd_*_v2 SET … (incl. new NK value + recomputed row_hash) WHERE id=:edit_id`, GUARDED by a pre-check that the new NK doesn't already exist on a DIFFERENT id (refuse with flash if it does; never silent-merge/dupe). This is a PHP change → an NK-field correction is NEVER JS-only. The pre-image audit lookup (pkg L1056, keyed on submitted_at+row_origin+suffix, NOT neb_batch) ALSO mis-snapshots under an NK-field edit: it finds the OLD row, logs `update` w/ old before_json against what is actually a fresh INSERT → incoherent audit. Fix the audit path in lockstep with the PK-UPDATE.
- **Closed-session lock (daily-shell forms):** the edit path loads the orig row's `session_id_fk`. If session-linked AND `op_sessions.status != 'open'` → REJECT ("correction verrouillée") — do NOT reopen a closed session. If `session_id_fk IS NULL` (legacy/backfill rows) → proceed UNLOCKED. Corrections never advance lifecycle: `session_advance_phase` is SKIPPED in edit mode (a correction is not a new event/step).
- **Firewall SKIPPED in edit render** (the start-phase eligibility gate is about *new* events; a correction to a past event must not be blocked by current vessel/cadence state).
- **Edit render shape:** banner + a NON-editable identity strip (the NK fields the operator must not change) + an editable `event_date` + the prefilled readings/dispositions. `auditFlags` gains `,correction` (e.g. `'web_entry,session_event,correction'`); `log_revision` is inherited from the handler.
- **VERIFICATION RECIPE (synthetic, self-cleaning — proven on fermenting):** insert a synthetic row, run the edit path twice: (1) preserved-submitted_at path → asserts exactly **1 row UPDATEd** (id stable, only the edited field changed); (2) deliberately-wrong fresh-submitted_at path → asserts **2 rows** (duplication = demonstrates the guard's necessity); then DELETE the synthetic rows (assert COUNT=0). ZERO real-data mutation. This is the standard proof the preserve-submitted_at guard holds before declaring done.
- **`event_date` has NO COGS/occupancy impact** on the bd_*_v2 family (the occupancy/cost anchors are elsewhere) → editing it in a correction is safe. `corrections_policy=allowed` for `bd_fermenting_v2` (check `schema_meta.corrections_policy` per table before enabling edit on a new form).
- **⚠️ EDIT PREFILL MUST NOT HYDRATE FROM THE LIVE CANDIDATE PICKER (RULED 2026-06-04, pkg `?edit=6737`; same lesson as racking edit).** In edit mode, identity (`neb_beer/neb_batch/contract_*/recipe_id`) + the SKU mosaic/format grid come from the STORED sticky payload (`PF_EDIT_STICKY_HEADER`), NEVER from a live tank-card's `dataset`. **Packaging bug:** `packaging-form.js` edit-restore (~L1244-1281) called `selectTank(foundCard)` when the stored source tank was still a live candidate; `selectTank` (L200) OVERWRITES `fldNebBeer/fldNebBatch/fldRecipeId` + re-renders the mosaic from `card.dataset` = the tank's CURRENT occupant. Because a NEWER lot was racked into that BBT, the card carried the new batch → editing the old row hydrated as the new lot, clobbering stored identity + format data (a COGS-bearing lot→SKU→BOM fact). FIX: delete the `selectTank` branch in edit mode; always run the direct-populate path (stored values win); a found card is visual-only (`--selected` class, no dataset read); **explicitly call `renderSkuMosaic(stickyH.recipe_id_fk)` in that path** (it was only ever called by `selectTank`) so the mosaic shows the STORED recipe BEFORE the format-restore loop. Same discipline for the `hors_process` round-trip (already correct: set hidden value, do NOT dispatch the checkbox change handler — that swaps the candidate list).

- **IN-PLACE CONVERSION sub-pattern (new 2026-06-03, fermenting cold-crash):** a correction can also CONVERT an event's type in place (e.g. non-ColdCrash → ColdCrash via a checkbox), not just edit fields.
