## ⚠️ THE SILENT-UNIT-ERROR CLASS (load-bearing lesson — read before any WAC/reclassify call)
**Definition.** A delivery row stores `qty_delivered` as a **pack / sachet / unit / 100-g COUNT** (NOT the canonical mass/volume) with a *compensating per-pack price*, so the invariant `qty×price = total_original` **HOLDS** → the row is INVISIBLE to the invariant (Pass-1) scan. It is detectable ONLY via: (a) per-category WAC/eff-price plausibility outliers, (b) catalog-divergence ratio far from the FX baseline, or (c) reading the source invoice. Same class as GALAXY/SIMCOE-105/WATER_DEMIN (migration 162) and PEACH_TEA + MOSAIC id=104 (migration 164).

**THE HARD-WON INSIGHT (Mosaic, 2026-05-27) — "expensive ≠ different product."** A WAC ANOMALY that *looks like* "a different / premium product" can be a UNIT BUG masquerading as a premium. The operator TWICE believed HOPS_MOSAIC id=104 (eff 137/kg) was an expensive Cryo Mosaic from "Hoppy People" and asked to reassign it to HOPS_MOSAIC_CRYO (ref_mi.id=54, which EXISTS but has 0 deliveries). The source invoice (Bière Appro **8389**, doc_invoices id=45, file_id 46) read plain "Mosaic, Yakima Chief, **Sachet 5 kg**, AA 11.7%" — a COMMODITY pellet; the 137/kg was the artifact of **2 sachets stored as qty=2** (2×5kg=10kg @ 29 EUR/kg). We nearly reassigned the MI on a CONTRADICTED premise. **RULE: a WAC outlier is a UNIT-BUG suspect FIRST, a product-difference SECOND. READ THE INVOICE before reclassifying any MI.** The PM HELD the reassignment — that was correct; reassigning would have corrupted both MIs.

**DETECTION SIGNALS to flag:** (1) `qty_delivered` is a small integer ({1,2,3,4,5,6,10,12,20}) for a mass/volume MI → likely pack/sachet count; (2) eff price (`total_chf/qty_delivered`) far outside the category band; (3) catalog-divergence ratio far from the FX baseline (≈0.945 for EUR MIs, ≈1.000 for CHF MIs — see below); (4) `details`/`notes_raw`/`ingredient_raw` containing "sachet / pcs / pack / 100 g / bidon / big-bag / ton / bouteille / 5 kg / L / 1Kg / 2Kg". **CORRECTION PATTERN:** read invoice → extract pack_size + billed unit → `qty_canonical = count × pack_size`, `unit_price = total_original / qty_canonical` (inverse-scale), **`total_original` UNCHANGED** → seed `ref_mi_invoicing_units` (pack_size in the MI's canonical unit). Never alter `total_chf` (invoice truth, already FX-applied).

## RM STOCKTAKE INPUT FLOW + how to CORRECT a mistyped count (canonical, verified live 2026-06-08)
**Two-table model.** Operator-facing UI = `public/modules/form-rm-stocktake.php` (gated `require_page_access('saisies')` = operator role; `?period=YYYY-MM`, defaults `date('Y-m')`; ref_pages: `warehouse`/`rm-comparison` are the logistics surfaces, the saisie form itself sits under `saisies`). It is a **per-pallet LINE-ENTRY form**: each physical pallet/count → one row in **`inv_rm_stocktake_lines`** (`mi_id`+`mi_id_fk`, `period`, `qty` DECIMAL(14,3), `is_active`, nonce `row_hash`; multiple lines per (mi,period) by design — NO mi/period UNIQUE). Add/delete is async JSON: `POST /api/rm-stocktake-line-add.php` / `POST /api/rm-stocktake-line-delete.php` (delete = SOFT, `is_active=0` + tombstone audit). **Every add/delete calls `rm_recompute_rollup()`** (`app/rm-stocktake-rollup.php`) which SUMs active lines → upserts **`inv_rm_stocktake.counted_qty`** (the canonical per-MI summary, UNIQUE `(mi_id,period)`, `final_qty = COALESCE(counted_qty, expected_qty)` STORED-generated — this is what COGS/RM reads consume). NULL-vs-0 invariant: 0 active lines → counted_qty=NULL (falls back to expected); ≥1 line → SUM. The summary row is NOT hand-edited via this UI — it is always a rollup of the lines.
**⚠️ Legacy periods are line-LESS.** Pre-line-table periods (e.g. 2026-04, 2025-12) have a DIRECT `counted_qty` in `inv_rm_stocktake` with NO backing lines (source `counted`/`operator-manual`). Only 2026-05+ (source `web-form`) is line-driven. So the correction path differs by period: line-driven → fix via the LINES UI; legacy direct → migration/direct UPDATE of `inv_rm_stocktake` (snapshot+log_revision).
**CANONICAL CORRECTION of a mistyped line (operator-self-serve, NO SQL, NO migration, NO reopen):** there is **NO month-close/lock table** — periods are never "locked", a correction needs no reopen. The fix is pure UI: open `form-rm-stocktake.php?period=YYYY-MM`, find the MI, **DELETE the wrong line, ADD the correct one** — each click auto-recomputes the rollup. Preferred over any SQL because it routes through the audited add/delete endpoints (CSRF + log_revision + soft-delete tombstone) and self-corrects `counted_qty` exactly. Use direct SQL ONLY for a legacy line-less period. WORKED EXAMPLE (2026-06-08, "Pack Découverte" PKG_PACK_DEC id 167 / fk 167, period 2026-05): summary row id=419 counted=7600 = lines 3600+520+600+960+960+960; line id=82 qty=3600 is the typo (extra zero, should be 360) → delete line 82, add a 360 line → rollup → counted_qty becomes 4360 (360+520+600+960+960+960). NB the 960 appears 3× (ids 85/267/272) — confirm with operator those are 3 genuine pallets, not an accidental re-entry, BEFORE he assumes only the 3600 is wrong.

## THE CANONICAL UNIT CONTRACT (per category)
Canonical inventory + cost unit:
- **Malt = kg** (ton/big-bag billed → normalize to kg). **Hops = kg** (input in g on the brew form; priced/stored kg). **Yeast = kg preferred** (currently MIXED g/kg — see drift below). **Process Chemical = kg** solids / **L** liquids (ml→kg needs density). **Brewing Adjunct = kg. Brewing Mineral = kg. Packaging = unit/each.** Liquids-by-volume (CO2/water/gas) = **L** (bottle → normalize to L).
- WAC invariant: `Σ(qty×price)/Σqty` is only correct if EVERY row's qty is in the canonical unit. Two error classes corrupt it: (1) gross-misscale (qty & price in different units → `qty×price≠total`); (2) silent-unit (internally consistent but qty in a non-canonical unit → invariant HOLDS, undetectable by Pass 1).
