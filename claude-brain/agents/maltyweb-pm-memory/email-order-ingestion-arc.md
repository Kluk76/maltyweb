## 🟢 RATIFIED 2026-06-15 — CANONICAL PRODUCER MODEL = MODEL B (validation-first, NO premature FK). PM ruling.
**The question:** where does a parsed email-order live at PARSE time? **RULING: Model B confirmed** — and it is not a free choice, the LIVE schema FORBIDS Model A for any unresolved order (PM-verified live 2026-06-15).
**Live constraints verified (`SHOW CREATE TABLE ord_orders`):**
- `requested_date` is **`NOT NULL` (column constraint, not a CHECK)** — cannot be deferred without a value.
- There is **NO `customer_id_fk NOT NULL` CHECK.** The ONLY CHECK = `ord_orders_chk_exactly_one_party`: `order_type='customer'` ⟹ `customer_id_fk NOT NULL` (+ internal_channel NULL) **XOR** `order_type='internal'` ⟹ `internal_channel NOT NULL` (+ customer_id_fk NULL). `order_type` is itself `NOT NULL`. So a *customer* order genuinely cannot exist without the customer FK.
- `review_status` ENUM = `('none','pending','accepted')`; `source` ENUM has `'email'`; `source_ref` UNIQUE (`uniq_ord_source_ref`); `source_email_id_fk` FK→doc_email_messages(id) ON DELETE SET NULL. All as-built.
- **Net: a hints-only `ord_orders` row is impossible** (NOT NULL requested_date + the party CHECK both force resolution at INSERT). Model A ⟹ guess the customer FK (never-guess violation) OR hard-fail on every sender-pin miss (drops orders). Model B is the honest shape mig 370 was built for.

**THE CANONICAL MODEL (Model B) — precise contract:**

NO customer/sku/date written as FACT — hints are strings/ints in parsed_json, never FKs.**

## RECOMMENDED ARCHITECTURE
- **Where the code lives:** STRANGLER RULE → prefer **maltyweb + MySQL + cron**, NOT a new standalone Node script. The mailbox poller is a Python connector on the VPS (sibling to the BC connectors `scripts/python/ingest_bc_*.py` — same cron pattern, same `config/*.env` creds model) that fetches new messages, dispatches to per-sender parsers, and writes `ord_orders` + a raw-email cache table. Parsers: a **PHP or Python per-sender registry mirroring `lib/invoice-parsers/`** — `matches(ctx)→bool` + `parse(body)→{customer, lines[], requested_date}`. PM leans **Python** (the connector is Python; keeps mailbox+parse+write in one VPS process; the JS invoice-parsers stay JS but the PATTERN is copied, not the code). Validation UI = maltyweb PHP.
- **Email body, not OCR:** email orders are text/HTML bodies (or attached PDF/xlsx). Text bodies → regex/structured parse directly (no OCR). PDF/xlsx attachments → reuse the OCR/xlsx path. The parser `ctx` = `{fromAddress, subject, bodyText, attachments[]}` (analogue of the invoice `ctx`).
- **Resolution:** customer via `ref_customers` (exact email→customer map first — store a sender-email→customer_id pin, analogous to `ref_mi_aliases`; then name match; else review). SKU via `ref_sku_aliases`/`ref_skus` (reuse the `(Item_No,UoM)` resolver pattern from BC Phase-1). Confidence gate: all lines resolve ≥ threshold + customer resolved → `review_status='accepted'`-eligible; else **`review_status='pending'`** (= awaiting logistics validation — see RECONCILIATION A; there is NO `needs_review`/`auto` value). Unparseable sender (no `matches`) → review bucket with raw body, NEVER guess.
- **Idempotency/dedup:** key on **RFC822 `Message-ID`** (stable per email) → the `doc_email_messages.message_id` UNIQUE col (AS-BUILT) and `source_ref='email:<message-id>'` on `ord_orders`. ## 🟢 PARSER PLAN — THREE-LAYER DETERMINISTIC STACK (Kouros approved 2026-06-15, "parse the BODY without an LLM"). NO LLM, EVER.
Kouros asked "can we parse the email BODY (not just PDF attachments) without an LLM?" → YES. The registry runs a 3-layer deterministic stack; uncertain → human, never guessed.
- **Layer 1 — per-sender templates.** Same doctrine as the PDF/invoice parsers, applied to `body_text`/`body_html`: HTML-table DOM walk OR aligned plain-text columns. `matches(ctx)→bool` keyed on from_address/subject; `parse(body)→ParsedOrder`. Built against Kouros's real per-sender samples.
- **Layer 2 — GENERIC vocabulary fallback (NEW).** Runs ONLY after all per-sender parsers decline. Matches the email body against our CONTROLLED product vocabulary loaded from CANONICAL tables (`ref_skus` / `ref_recipes` / `ref_beer_types` + alias tables — **NEVER hardcoded beer names**) via the existing fuzzy matcher (entity-resolve token-sort ≥ 0.85); then extracts the adjacent quantity+unit via a deterministic grammar (fûts / cartons / bouteilles / cans / "2x" / "20L" / "6-pack") and a date-phrase parser (FR weekday / dd-mm, honoring `system_setting` date_parse_dayfirst). **Conservative confidence gate: emits a ParsedOrder line ONLY for a line that maps to a KNOWN SKU with an UNAMBIGUOUS qty; everything else → review bucket.** Registered LAST. Tune against real samples when they arrive.
- **Layer 3 — confidence gate.** Anything below the bar → the review bucket (the existing validation-first surface = `doc_email_messages.parse_status` + the P3 UI). "No LLM" = uncertain orders go to a human, never guessed.
- **🔑 Registry dispatch refinement (RECORD):** a parser's `parse()` may return **`None` = "I decline, try the next parser"** (so a per-sender parser that matches the SENDER but can't read THIS body falls through to layer-2); **`raise` = error** (→ parse_status='error'); **exhausted / all-None = no_match** (→ parse_status='no_match'). This generalizes the invoice-parsers first-match-wins into match-AND-parseable-wins; record it as the email-registry contract. (Substrate already built `0c0bb5e`: `email_parsers/__init__.py` registry + dispatch — extend it to this 3-layer + decline-semantics; the example_template is the L1 reference; layer-2 is a new generic parser registered last.)
