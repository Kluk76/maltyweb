## CARDINAL VERDICT
**A mother session is a multi-day BATCH-LIFETIME envelope that GROUPS daily child sessions. Daily children stay the canonical operational shells; the mother adds a long-running parent that travels with the batch (recipe + batch number) across vessels, across days, across phases, across operators. The mother stores NO derived facts** — recap volumes, total losses, packaged HL, ETAs all DERIVE from the children + the event tables, exactly like the daily session model. Mother = lifecycle envelope at the batch grain; children = lifecycle envelope at the day grain; events = truth. Both layers are envelopes, neither stores derived state.

## WHY IT EXISTS (operator framing)
The daily 3-phase shells answer "what happened in this vessel today" — they are operationally correct for racking, fermenting, packaging, brewing taken one day at a time. But a brewing batch's full story is 4-12 weeks across multiple vessels (mash → boil → cool → CCT fermentation → BBT cellaring → KZE → packaging line → close). The operator wants ONE card per batch that travels the whole arc, showing every daily child session as a sub-event, with the batch's identity (recipe + batch number) as the through-line. Without it, "give me everything on Embuscade batch 245" is a 4-table query across daily sessions, not a single board view. The mother shell makes the BATCH a first-class lifecycle object.

## DATA MODEL — additive, ZERO daily-pilot rework — AS-BUILT (mig 215, commit `96ea0d4`)
**LANDED 2026-05-29.** One ENUM value + one FK column + one optional column + a filtered UNIQUE + a CHECK on `op_sessions`. The mig 192 schema already had the dormant `parent_session_id_fk BIGINT UNSIGNED NULL` — wired up by Phase 1.

- `op_sessions.form_type` ENUM extended with `'batch'` (additive — every existing daily session keeps its racking/fermenting/brewing/packaging value). ✅ verified live.
- Mother row shape (`form_type='batch'`): `recipe_id_fk NOT NULL`, `batch NOT NULL`, `vessel_kind = NULL`, `vessel_number = NULL`, `phase ENUM('start','in_progress','end','closed')` (existing — reused with batch semantics: `start`=brewing in flight, `in_progress`=cellaring/racking/packaging in flight, `end`=last packaging closed/recap pending, `closed`=operator/auto-confirmed all done), `status ENUM('open','closed','abandoned')` (existing — reused).
- `op_sessions.merged_into_session_id_fk BIGINT UNSIGNED NULL` (FK → `op_sessions.id` ON DELETE SET NULL) — populated ONLY when a source mother is absorbed into a surviving mother at blend time. Self-referential FK, mirrors the dormant `parent_session_id_fk` pattern (additive, non-breaking, NULL = the row is not a merged-source). ✅ live.
- `op_sessions.blend_share_pct DECIMAL(5,2) NULL` (OPTIONAL) — records the % share of the source mother that joined the surviving mother at merge time. NULL by default; populated when operator captures the ratio. ✅ live.
- **DEVIATION (planned vs as-built — UNIQUE):** the spec planned a generated VARCHAR(64) `active_batch_key` virtual col with `CASE WHEN form_type='batch' AND status='open' THEN CONCAT(recipe_id_fk,'#',batch) ELSE NULL END` + UNIQUE. **AS-BUILT** = `open_mother_key CHAR(1)` VIRTUAL GENERATED + UNIQUE `uniq_active_mother (recipe_id_fk, batch, open_mother_key)`. The CASE-in-functional-index pattern is NOT supported in MySQL 8 — the virtual CHAR(1) column gating uniqueness is the canonical pattern (NULL-bypass on the composite UNIQUE achieves "one open mother per (recipe, batch)"). Spec §7.4 supersedes itself.
- **CHECK** `chk_op_sessions_blend_share_range` (0..100 or NULL, table-prefixed per MySQL-8 gotcha; schema-unique vs 3 existing `chk_op_sessions_*` — abandon_pair / closed_pair / vessel_xor). ✅ live. The planned `chk_op_sessions_batch_shape` ruling about mother-doesn't-pin-to-vessel is structurally enforced by the auto-link resolver (mother is never inserted with vessel_kind/number set) — promote to a CHECK later if needed; not required Phase 1.
- NEW filtered-UNIQUE for the active-batch invariant: only ONE mother per (recipe_id_fk, batch) may be `status='open'` at once. MySQL-8 pattern = generated col + UNIQUE: `active_batch_key VARCHAR(64) GENERATED ALWAYS AS (CASE WHEN form_type='batch' AND status='open' THEN CONCAT(recipe_id_fk,'#',batch) ELSE NULL END) VIRTUAL` + `UNIQUE KEY uq_op_sessions_active_batch (active_batch_key)`. NULL keys are unique-friendly in MySQL 8 → closed/abandoned/merged mothers don't block re-opening; multiple sources can have been merged into one survivor without violating uniqueness.

**Daily children stay AS-IS.** No schema change to `bd_racking_v2` / `bd_fermenting_v2` / `bd_packaging_v2` / `bd_brewing_*_v2` / `bd_cip_events`. The daily-child `op_sessions` rows gain `parent_session_id_fk = <mother.id>` (already an existing nullable col since mig 192). Auto-link runs in `session_open()` for daily children: look up the OPEN mother by `(recipe_id_fk, batch, form_type='batch', status='open')`; if found, set child's `parent_session_id_fk`; if not found, CREATE the mother row first (auto-spawn at first brewing child open) then link. Idempotent: re-opening a daily child of an already-linked batch just hits the existing mother.

## CLOSE PREDICATE — Q1 OPERATOR OVERRIDE (different from expert default)
**The mother closes on the "cuve vide" button in the packaging sheet, NOT on a per-mother "Clôturer le lot" CTA.** Operator chose the simpler, more naturally-occurring trigger: when the operator marks a BBT (or CCT in rare cases) as empty at end of a packaging session, the system identifies which mother(s) had liquid in that vessel — via `bd_racking_v2` destinations + the existing `TankSimulator` filtered by `op_sessions.parent_session_id_fk` — and auto-closes them.

- **Trigger surface:** the packaging form's "Cuve vide" button (operator-confirmed at the OPERATIONAL moment "I just emptied this BBT"), NOT a separate mother-level UI prompt.
- **Resolver chain:** `cuve_vide_pressed(vessel_kind, vessel_number)` → query all OPEN mothers whose latest racking destination = this vessel (via `bd_racking_v2.dest_vessel_kind`/`dest_vessel_number` joined through `parent_session_id_fk`) → for each: if NO open packaging child remains AND the BBT is now empty → `finalize_mother(mother_id, reason='vessel_emptied')`.
- **Auto-close FROM mother POV, operator-confirmed AT THE VESSEL LEVEL.** This is the elegant split: the operator never has to remember "is this batch finished?" — they answer "is this BBT empty?" which is the question they are answering anyway at end-of-packaging.
- **Dialog on close** (Q2 expert default kept): close UI shows volume summary (brewed HL / packaged HL / loss %) + a `raison ENUM('emballé','jeté','vendu moût','encore en cuve')` dropdown. The reason is recorded in the `recap_acknowledged` payload on `op_session_steps` (existing step_type, no new ENUM value).
- **`encore en cuve` case:** operator dismissing the auto-close (they pressed "Cuve vide" but the batch lives on in a different vessel) → mother stays `open`, the "Cuve vide" press records as a step but does NOT advance mother phase. Refuse-don't-NULL discipline at the close boundary (mother stays open rather than silently close on contradictory state).

## BLEND / MERGE SEMANTICS — Q3 OPERATOR OVERRIDE (much simpler than initial model)
**At blend moment: ONE mother SURVIVES, the OTHER is FORCE-CLOSED OFFICIALLY and TAGGED as merged into the survivor.** This is the operator's much simpler ruling — supersedes the earlier "blend mother + bridge table + status='merged'" sketch.

- **Survivor pick:** at blend UI, operator picks which mother SURVIVES (the destination). "Généralement plus ancien mais pas forcément" — operator-driven, no auto-rule.
- **Source closure:** the OTHER mother is force-closed officially: `status='closed'`, `phase='closed'`, `merged_into_session_id_fk = <survivor.id>`, `blend_share_pct` optionally populated. NO new ENUM value (no `status='merged'` — `closed` covers it; the differentiator is `merged_into_session_id_fk IS NOT NULL`).
- **Surviving mother continues operationally** with its packaging child(ren). It can later absorb additional source mothers in subsequent blends — multi-source absorption is a 1:N pattern (`SELECT * FROM op_sessions WHERE merged_into_session_id_fk = <survivor.id>` returns ALL absorbed sources). No bridge table needed.
- **Closed source stays VISUALIZABLE WITH the surviving mother.** Drill-in on the survivor shows the absorbed source(s) as layered cards. Surface UX: "EMB 245 fusionné dans EMB 244 @ 40%" badge on the survivor; drill-in shows each merged source. Source mothers are "absorbed" — accessible via drill-in but NOT on the active board.
- **Why simpler than the initial model:** no bridge table; no new ENUM value; one new column + one optional column; mig 204 stays small. The asymmetry (survivor + N absorbed sources, all keyed by one self-FK) is the entire complexity. Discipline: never invent a blend-shaped third mother — that would be a parallel-store smell vs the survivor.

## DIVERGENCE FLAGS
- 🔴 **NO derived-current-state column on the mother.** Recap volumes, total losses, ETAs all derive from children + events. The same anti-pattern session-model-arc.md guards against at the daily layer.
- 🔴 **Mother NEVER pins to a vessel.** The CHECK enforces it. A vessel is a transient location for a mother's children, never an attribute of the mother itself. Putting a vessel on the mother row is the classic parallel-store smell.
- 🔴 **`merged_into_session_id_fk` survivor pattern, NOT a `status='merged'` ENUM value.** A merged source is `status='closed'` with a non-NULL self-FK — the FK presence is the differentiator. Inventing a `merged` status creates ambiguity (closed vs merged-closed) without a benefit.
- 🟠 **Auto-link in `session_open()` is the only acceptable wiring point.** A separate manual "link this child to a mother" UI is the parallel-store smell — children should NEVER have a mother chosen post-hoc by an operator (Q6 forward-only ruling enforces this at the historical-data layer too).
- 🟠 **"Cuve vide" trigger MUST verify the vessel is actually empty.** Refuse-don't-NULL at the close boundary: if the trigger fires but `TankSimulator` says the vessel still holds liquid, the mother stays `open` and a step row records the contradiction. Never silently close on contradictory state.
- 🟠 **Filtered UNIQUE on (form_type='batch', recipe_id_fk, batch) for status='open'** prevents a 2nd mother row colliding on the same active batch. Achieved via generated-col + UNIQUE per the MySQL-8 pattern in op-sessions-schema-spec.md §Q2. Two open mothers for the same (recipe, batch) IS a divergence smell — the DB rejects it.
- 🟡 **Vessel SVG rework is IN scope, NOT a separate track.** Concept C diorama requires it; do not ship the board without it. The "Vue cuves" toggle on the daily Journal de bord (existing badge cards) inherits the rework when it lands.
- 🟡 **EXPÉDITION zone is a v1 placeholder.** Render it as dashed/grey on every closed mother card so the visual contract is set for the Stage-3 activation; do NOT wire any dispatch logic.
- 🟡 **ETA + heartbeat compute on read.** Pure derivations from closed mothers / event timestamps. NO cache table, NO materialization — session-model-arc.md §READ-CADENCE poll-15s pattern applies on the active board.
