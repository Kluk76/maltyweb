### 🔴 DURABLE KPI/VIZ LESSONS (two post-ship bugs Kouros caught on live pixels — fold into ALL future widget builds)
- **LESSON 1 — `grouped_bar` consumes `result.breakdown[]`, NOT `series[].groups[]`.** `renderKpiGroupedBar()` in `public/js/kpi-charts.js` reads **`result.breakdown[]`** where each row = `{key, label, value, meta:{prior_year}}` → solid bar = `value`, ghost bar = `meta.prior_year`, **shared max-scale across all rows**. A handler that emits `series/groups` for `viz_type=grouped_bar` renders **headline-only** (the empty-breakdown branch fires, no bars). **The widgets first shipped emitting `series/groups` → bars were invisible.** Always shape grouped_bar payloads as `breakdown[]`. (Companion to the existing bar-viz rule: `renderKpiBar` reads `series[].period`; grouped_bar reads `breakdown[]`.)
- **LESSON 2 — `grouped_bar` shares ONE max-scale across every row.** Feeding absolute values of very different magnitudes (Semaine 348 vs Année 16 700 hl) makes the small bars invisible. **Feed a magnitude-independent measure** — here we used **% d'atteinte** per horizon, so all three rows are 0–100% on a common scale. Choose the row measure so rows are comparable, not raw quantities that span orders of magnitude.
- **LESSON 3 — extending a SHARED `KpcCharts` renderer must be additive / opt-in** so other trackers using it stay untouched. We extended `renderKpiGroupedBar` with two opt-in hooks: **(a) per-row unit override `b.unit || result.unit`** (lets one row read "%" while the tracker unit is "HL"), and **(b) custom neutral chip via `b.meta.chip_label`** (falls back to the computed YoY ▲/▼ chip when absent). **Verified unchanged:** #5 `production_by_beer_yoy` and #274 `units_by_sku_month` (both grouped_bar) render exactly as before. When you touch a shared renderer, list its other consumers and confirm them byte-stable before/after.
