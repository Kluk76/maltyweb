Do NOT overload `source='bc'` for maltytask-born-then-echoed orders (that's the echo-loop trap, see D).
  - **Corrections (Pepitium, B):** edit-in-place + FLAG, do NOT create a new line revision. ord_order_lines stays the live operational truth; the BC-line snapshot retains the divergent BC value; the `divergence_status` flag on the order says "maltytask ≠ BC, comptable correction required". Line-revision history lives in `ord_order_status_events`-style append log only if Kouros later wants an audit trail — NOT needed for v1.

**RULING — PHASE 2 WRITE-BACK ARCHITECTURE (decision 3), the hard parts:**
  - **(a) ECHO-LOOP:** maltytask creates order in BC → `*/15` reader pulls it back → MUST NOT duplicate. Mechanism = maltytask writes a maltytask-owned external ref into a BC field the reader keys on; reader's source_ref upsert (`bc:<No>`) is already idempotent on BC OrderNo, but the maltytask-born order has a LOCAL id BEFORE BC assigns a No. **Resolve by: maltytask-born order gets a provisional local source_ref (`mt:<localid>`); on successful BC create, store the returned BC No and REKEY source_ref → `bc:<No>` in the SAME write transaction; the reader then matches it as existing on next pull (UPDATE not INSERT).** Do NOT rely on External_Document_No for echo identity (see b).
  - **(b) 🔴 External_Document_No DUAL-USE COLLISION — FLAG:** Kouros ALREADY uses BC `External_Document_No` to distinguish SPLIT orders (same customer, different location). Writing maltytask refs into it would COLLIDE with that operational use. → Phase 2 must NOT hijack External_Document_No for the maltytask echo ref. Use a DISTINCT BC field (a custom field e.g. `MaltytaskRef`, or `Your_Reference`, or the Shopify-style `ShpfyOrderNo` pattern of a dedicated custom column) — confirm with Kouros which BC field is free. This is a hard prerequisite for D's create path.
  - **(c) IDEMPOTENCY + PARTIAL FAILURE:** every write carries the maltytask local id as the dedup key (the provisional ref); a create that times out but actually landed in BC must not double-create on retry → before create, GET BC by the maltytask ref; create only if absent. Wrap create+rekey in a local transaction; if BC create succeeds but local rekey fails, a reconcile pass (the reader) recovers by matching the BC ref back to the local provisional row.
  - **(d) CREDIT-MEMO (Pepitium):** create avoir (`salesCreditMemos`) for the wrong locked invoice + corrected re-invoice. Start with CREATE only (per Kouros: CREATE orders + credit memos; POST stays human in BC). The B-flag (`correction_compta_requise`) is the queue that feeds D's credit-memo generation.
  - **(e) BC API CHOICE:** standard `salesOrders` / `salesCreditMemos` POST (the standard Business Central v2.0 API) for CREATE — cleaner than OData web services for writes, supports the standard entity shapes. Reuse the S2S OAuth2 client-creds from `config/bc.env`. (The read side uses OData `SalesOrder`/`SalesOrderSalesLines` web services for richer fields; writes can use the standard API independently.) BC write permission = Kouros's BC-side permission-set change (SUPER role, Entra has API.ReadWrite.All) — he does it himself; we build assuming it lands. **Start with CREATE, NEVER POST (no posting/invoicing from maltytask initially — keep that human in BC).**

**RULING — PHASE-1 READER CHANGES when Phase 2 coexists (decision 4):**
  - Status-preserving upsert UNCHANGED in spirit (status operator-owned), but the reader must learn to RECONCILE maltytask-born orders: a pull row whose maltytask-ref matches a local `mt:`/rekeyed `bc:` row = UPDATE (the echo), never a new INSERT. → reader keys on the maltytask ref FIRST, then `bc:<No>`.
  - **Collision logic is NOT YET External_Document_No-aware (deferred per Phase-1).** With B/C/D: it STILL doesn't need External_Document_No for echo (we use a distinct maltytask ref, decision 3b). BUT — because External_Document_No legitimately splits same-customer/different-location orders — the reader must NOT treat two BC orders sharing a customer as one; it already keys on BC OrderNo (unique), so split orders stay distinct. **No change forced by B/C/D, but the deferred External_Document_No-awareness should be revisited IF Kouros ever wants the board to GROUP split orders — flag, don't build.**
  - `bc_completely_shipped → bl_printed` auto-advance: REMOVE it (ruling 4) as part of B; replace with reconciliation signal. Once removed, the reader never writes status at all — clean separation Phase 2 depends on.

**DIVERGENCE FLAGS (this program):**
  - **🔴 Echo-loop double-create** (D) — guarded by maltytask-ref-first reconcile + GET-before-create idempotency.
  - **🔴 External_Document_No dual-use** (D) — do NOT use it for echo identity; it's Kouros's split-order key. Use a distinct BC field.
  - **🔴 Double-deplete via new line statuses** (C) — fg-stock Step 4 + all ord_order_lines depletion reads MUST exclude non_livre/rupture; Opus independently verifies the no-deplete invariant (COGS/inventory-bearing).
  - **🔴 Breaking the LIVE Phase-1 reader** — B refactors the connector (snapshot + diff + remove auto-advance); regression-gate the status-preserving upsert (no unauthorized status writes, no spurious INSERT) before `--apply`. The reader is operationally hot (`*/15`).
  - **Parallel store** — `ord_order_bc_lines` (BC snapshot) is NOT a second copy of ord_order_lines; it is the retained BC-side fact for diffing. Keep them distinct-by-purpose; never let anyone "reconcile" them as one (they DIVERGE by design — that's the whole point of B).
  - **COGS coupling** — operational truth (ord_orders/lines, post-correction) drives fg-stock depletion + COGS; BC reconciled via avoir. A correction (Pepitium DGDF→DGDB) changes which SKU depletes → COGS-bearing → Opus verifies the corrected line depletes the right SKU.
