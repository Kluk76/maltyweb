## THE THREE DECOMPOSITION AXES (the framing)
A SKU's BOM decomposes along three axes. Two are wired; the third is this build.
1. **Financial (COGS/COP):** `ref_sku_bom` cost lines (additive across lines). LIVE.
2. **Volume (HL):** `ref_sku_bom.volume_hl` on the CONTAINER line only + `v_sku_volume` view (additive-cost-vs-OWNED-volume rule ‚Äî volume-dimension.md). LIVE (Phase A).
3. **Per-recipe HL SHARE ‚Üí ingredient attribution (THIS BUILD):** each pack decomposes into its share of HL *per recipe*, so underlying ingredient consumption is computable per recipe. Single-recipe packs = trivially 100% one recipe. **MIXED packs** (deco/collab samplers spanning multiple beers) split HL across recipes; ingredients attribute proportionally.

## THE DATA MODEL (DBA-grade, settled)
**One canonical component-map table = `ref_sku_composite_slots` (already exists).** A composite/mixed SKU is defined by its rows there: `(sku_id, recipe_id, units_per_recipe, member_format_id, slot_order)` + SCD2 dates. No parallel store.

**Per-recipe HL share is DERIVED, not stored** (refuse a parallel volume store):
- `slot_hl_for_recipe = units_per_recipe √ó hl_per_unit(member_format_id)` (the member's own container fill volume from the volume chain ‚Äî `ref_packaging_formats.hl_per_unit`, itself = units √ó `dbc_container_types.hl_per_unit`, zero-drift on standard formats).
- `pack_total_hl = Œ£ slot_hl_for_recipe` over the pack's slots. (For a mixed pack this is genuinely additive ACROSS DISTINCT MEMBER recipes/SKUs ‚Äî different from the within-one-SKU accessory-line non-additivity of axis 2.)
- `recipe_hl_share(recipe) = slot_hl_for_recipe / pack_total_hl`.
- **Ingredient attribution:** per-recipe consumption = `recipe_hl_share √ó pack_total_hl √ó ingredient_per_hl(recipe)` = simply `slot_hl_for_recipe √ó ingredient_per_hl(recipe)` (the share cancels ‚Äî attribute each member's HL directly to its own recipe's per-HL ingredient basis). Per-HL basis from the canonical observed-liquid source (`ref_recipe_profile_*` / `bd_*_v2` / `ref_recipe_ingredients` gap-fill ‚Äî observed wins).
- **BOM emission:** the composite compiler emits `bom_source='composite_liquid'` lines (one set per member recipe, qty = per-HL √ó slot_hl) + `composite_packaging` lines (the pack's own outer packaging via choices/bindings/template). `volume_hl` on composite liquid lines follows the additive rule across MEMBERS (each member line owns its member container volume).

**Rollup (axis 3 deliverable):** a per-recipe ingredient rollup is a VIEW over `ref_sku_bom WHERE bom_source IN ('liquid','composite_liquid')` GROUPED to (recipe, mi). Reuse the canonical (period √ó entity √ó measure) matrix template ONLY at the monthly layer (`volume_monthly` / a future `recipe_consumption_monthly`), keyed by period √ó recipe √ó mi √ó measure ‚Äî parallel to `cop_monthly`/`cogs_monthly` (all 3 exist as EMPTY shells; see landmine (a)). The per-SKU rollup itself is a thin view, not a matrix table.

- **Mixed-pack HL precision:** derive HL from the integer `units_per_recipe √ó member hl_per_unit` chain (zero-drift on standard formats); do NOT store a rounded share. DECIMAL(10,6) on any persisted volume_hl. The share cancels in attribution (attribute member HL directly) ‚Äî avoids rounding a ratio.

- **`-X` cage interaction (CRITICAL, cross-link packaging-bom-model/README.md MULTIPACK STOCK-PULL).** A multipack pulled from `-X` cage stock already has its bottle+cap+label cost booked ONCE in `-X`. Per-recipe attribution must NOT double-book: the bottle in a `-X` cage already belongs to ONE recipe (the `-X` is per-beer). So a `-X`-pulled multipack's per-recipe liquid attribution = a DRAW-DOWN from that beer's `-X`, NOT a fresh ingredient consumption. Only the UNBOXED path (acct-1080 priv√© OR Shopify-pickup) consumes loose units ‚Üí fresh attribution. **The composite decomposition (this build) computes the THEORETICAL per-recipe HL/ingredient map; the ACTUAL consumption routing (cage-draw vs fresh) is the Stage-2 multipack-stock-pull build.** Keep them distinct: axis-3 here = the map; Stage-2 = which physical stock the map draws from.
