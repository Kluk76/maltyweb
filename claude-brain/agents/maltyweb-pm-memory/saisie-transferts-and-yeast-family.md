## §OCCUPANCY-MODEL RULING (PM, 2026-05-27 — APPLIED EXACTLY in `fa6221c`, live-proven; kept for the rationale record)
> **STATUS: APPLIED.** Both candidate paths route through TankSimulator as ruled. Gated = SQL+garde HAVING+Q2 EXISTS → sim; hors-process = SQL (no HAVING/no EXISTS) → sim. Live proof: gated = 3 un-racked Zepp (CCTs 12/14/16); sim drops already-racked (hors-process 18→15). The Q1-Q4 reasoning below is the as-ruled record.

**Problem the operator surfaced:** the Phase-3 gated candidate SQL freed a CCT ONLY by a later brew into it ("no later brew" NOT EXISTS) — NO exclusion for a lot already RACKED OUT. So a transferred lot stays wrongly eligible in the post-rack/pre-next-brew window (doesn't manifest today only because the 3 current Zepp lots 210/211/212 aren't racked yet; racking tops out at batch 209).

**Lifecycle fact (VERIFIED):** TankSimulator (`app/tank-simulator.php`) models the FULL COOLING→RACKING→PACKAGING lifecycle, CCT INCLUDED. `processEvent` COOLING (L346-370) FILLS the CCT (= the pre-rack fermentation occupancy this form needs); RACKING (L372-376) `$cctState[$e['cct']]=null` EMPTIES the source CCT on rack-out — exactly the missing exclusion. So the sim is NOT post-rack-only; it is the authoritative pre-rack CCT-occupancy source too. `form-packaging.php` already treats it as the single authority (L492 run(); L512 drops candidates whose tank is null in sim).

**bd_racking_v2 = DESTINATION only (VERIFIED schema):** `racking_destination_type` ENUM(BBT,CCT,YT) + bbt_number/cct_number/target_tank_raw; NO source-CCT column. Matched to a lot by neb_recipe_id_fk+neb_batch (or contract_*). NEVER read bd_racking_v2 to determine what is IN a CCT (the file's own L256-257 comment already knew this) — only as a cheap "has-been-racked" pre-filter SIGNAL.

**THE CANONICAL MODEL (architecture = identical to form-packaging): coarse SQL pre-filter → TankSimulator authority.** One occupancy model, shared by both forms. Pure SQL is NOT the authoritative occupancy source — adopting an SQL-only rack-out exclusion = a 2nd divergent occupancy model in SQL (forbidden parallel-truth smell).

**Q1 — sim or purpose-built SQL for pre-rack? → SIM is authoritative.** SQL stays as the cheap pre-filter that carries the cold-crash + garde join the sim doesn't compute, then candidates run through `(new TankSimulator($pdo))->run(today)` and any whose source CCT is null/empty/holds a different (beer,batch) get dropped. Mirror form-packaging L495-524 filter helper — do NOT re-implement differently.

**Q2 — "any racking row ⇒ excluded"? → OK as a PRE-FILTER on the time-gated path ONLY; NOT the authoritative gate.** Defect 1: PARTIAL racks (rack half, leave residual fermenting, rack remainder later) have a racking row while still physically in the CCT → "any racking row ⇒ exclude" wrongly hides a still-transferable lot; the sim handles this by volume, an EXISTS check can't. Defect 2: it's a 2nd occupancy model in SQL. Ruling: add the `NOT EXISTS (SELECT 1 FROM bd_racking_v2 rk WHERE rk.neb_recipe_id_fk=bfw.recipe_id_fk AND rk.neb_batch=bfw.batch AND rk.is_tombstoned=0)` to the GATED query as a cheap pre-trim, THEN let the sim be the tiebreaker/truth.

**Q3 — CCT→CCT blend re-racks? → OUT OF SCOPE for the time-gated path; hors-process ONLY. Do NOT build current-CCT-after-blend tracking in this build.** A blended lot (a) already has a racking row [Q2 pre-filter hides it] and (b) its `bd_brewing_brewday_v2.cct` points at the ORIGINAL brew CCT not the blend-destination → neither SQL nor sim can locate it in its current vessel (sim's batchCCT map L482-503 is brewday-keyed; RACKING clears the original CCT, never populates a destination CCT as a new fermentation occupant). There is NO canonical way to track current-CCT-after-blend today. Inventing one (reading destination_type='CCT'+cct_number back into occupancy) = occupancy derivation built on a guess about the blend lot's identity + garde clock = refuse-don't-guess violation. Reachable only via hors-process (manager + reason). If blend-then-2nd-rack becomes recurring, it needs a DELIBERATE model extension (source-CCT column on racking + sim support for CCT-as-rack-destination) — not smuggled into this gate. LOG as backlog gated on real workflow need.

**Q4 — does hors-process need the same exclusion? → It needs the SIM, but NOT the Q2 EXISTS.** Hors-process relaxes the garde TIME gate, not physical reality. A fully-racked-out lot is gone → must NOT appear even hors-process (the SIM enforces this, its CCT is null). BUT the Q2 EXISTS is too blunt for hors-process: a partial-rack lot (racking row exists, beer still in CCT) is EXACTLY the off-nominal case hors-process exists to surface — hard-coding "any racking row ⇒ exclude" would hide it. Ruling: **Time-gated path** = SQL (cold-crash join + garde HAVING + no-later-brew + Q2 EXISTS) → SIM. **Hors-process path** = SQL (no-later-brew, NO garde HAVING, NO Q2 EXISTS) → SIM. The sim is the common physical-reality layer for both; the EXISTS pre-filter belongs ONLY on the time-gated path. Mirrors form-packaging's override list (L526+) built from sim-occupied tanks, not the racking pre-filter.

**DISPATCH (Sonnet, equip coder+sql; RULE-2 before commit — check: sim filter applied to BOTH lists, no divergent occupancy logic, no helper scripts left):** (1) keep coarse SQL; add Q2 NOT EXISTS to the GATED query only (after garde HAVING); (2) run `(new TankSimulator($pdo))->run(today)` and filter BOTH candidate lists through sim CCT state (drop null-CCT or mismatched (beer,batch)) — lift the helper from form-packaging L495-524; (3) hors-process query drops garde HAVING, keeps no-later-brew, OMITS Q2 EXISTS; relies on sim for physical reality; (4) document the CCT→CCT-blend limitation in the file header + backlog. **[DONE — shipped exactly as dispatched in `fa6221c`.]**
