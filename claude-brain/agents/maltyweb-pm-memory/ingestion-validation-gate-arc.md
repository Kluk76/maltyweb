### 🔴 SECOND commit-flow bug — the "4-min buffer" mystery: commit-completion poll was structurally blind (FIX `8b697a1`, deployed)
- **Symptom:** EVERY invoice commit spun the À-valider card to the 4-min poll cap even though the commit actually SUCCEEDED. (Distinct from the 2026-06-08 exec-bit outage above — that wrote nothing; this writes correctly but the UI never sees `done`.)
- **Misdiagnosis (corrected):** earlier in session assumed tsx worker cold-start (~1-2 min). **FALSE** — measured a real cold-start commit end-to-end at **0.85s** (full tsx load + pipeline + MySQL + commit-check). Worker speed was NEVER the problem. (This also retro-corrects the 2026-06-08 "tsx cold-start can exceed 2-min poll cap" framing that motivated the 2→4 min cap bump in invoice-validate.js — the cap was never the real constraint.)
- **ROOT CAUSE:** `public/api/upload-status.php` only resolved `$doc_invoice_id` INSIDE a guard `if ($drive_file_id !== '' && !in_array($pipeline_status, ['failed','processed']))`. But a **staged invoice is ALREADY `pipeline_status='processed'` before the operator commits** → block skipped → `$doc_invoice_id` stayed null → the `commit_status` block (gated on `$doc_invoice_id !== null`) never ran → endpoint returned `commit_status: null` → `invoice-validate.js` poll read null as "still pending" → looped to POLL_MAX_ATTEMPTS.
- **FIX:** guard changed to `$pipeline_status !== 'failed'` so `$doc_invoice_id` resolves even when already 'processed'. Inner status-derivation branches are independently guarded by `if ($new_status !== 'processed')`, so re-entry is a no-op apart from populating `$doc_invoice_id`. Verified: the 3 committed invoices now return `commit_status=done`; commits flip green in ~2s.
- **🔑 DURABLE LESSON:** the commit poll DEPENDS on `$doc_invoice_id` resolution. **Any change to upload-status.php's status-derivation guards MUST preserve `$doc_invoice_id` population for already-'processed' invoices** — or the green-card poll goes blind again.
