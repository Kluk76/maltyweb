**MODEL VERDICT: EVENT-SOURCED, CONFIRMED.** Record `transfer_interrupted` + `beer_lost` + the existing `racked_vol_hl`/`blend_hl` on the row; the simulator RE-DERIVES occupancy from the event stream. **1. TankSimulator interpretation (the two new cases vs today's full-empty):**
   - **Today (normal rack):** source CCT → null (full empty); dest BBT += netRacked (+blend rescale). Volume-blind on the source side.
   - **(a) interrupted, NOT lost** = a PARTIAL rack. Source CCT must KEEP its remainder. **Sim change: on RACKING, instead of unconditional `$cctState[cct]=null`, DECREMENT `$cctState[cct]['volume_hl'] -= racked_vol`; if remainder ≤ EMPTY_THRESHOLD → null (fully emptied after all), else keep occupant (same beer/batch, reduced volume).** Dest BBT fills as normal (+racked_vol+residual). This is the deterministic full-vs-partial test (see point 4). NB the loss constant: keep `RACKING_LOSS_HL` applied to the DEST fill (netRacked) as today; the SOURCE decrement uses the gross `racked_vol` (what left the tank).
   - **(b) beer_lost** = racked_vol DESTROYED, NOT deposited in dest. **Sim change: process the SOURCE side exactly as the interrupted case (full-empty if racked_vol≥source total, else decrement) BUT SKIP the dest-BBT fill entirely (no `$bbtState[bbt]` write, no blend).** If residual_lost too (see point 5), the dest BBT's pre-existing residual is also destroyed → set that dest BBT null/decrement. beer_lost is a SINK: volume leaves the source, never lands.
   - One sim, both candidate paths still route through it (the §OCCUPANCY-MODEL RULING stands). The flags only change the RACKING branch's arithmetic.

**2. Eligibility gate predicate (the partial-rack-stays-eligible problem).** Current gate = coarse SQL (cold-crash+garde+no-later-brew) + Q2 `NOT EXISTS(bd_racking_v2 for lot, is_tombstoned=0)` pre-filter (gated path only) → TankSimulator authority. **The Q2 EXISTS pre-filter is now WRONG for interrupted-not-lost** (writes a racking row, but source still holds beer → must stay eligible). **RULING: the SIM is already the correct authority — let it decide, and FIX Q2 to not over-trim.** Two coherent options, prefer (i):
   - **(i) PREFERRED — drop the blunt Q2 EXISTS; rely on the (now volume-aware) sim.** Once the sim decrements instead of full-empties on a partial/interrupted rack, a partially-racked CCT that still holds beer remains occupied in `$simState['cct']` → the `simFilterCct` (L446-459, drops null-CCT candidates) keeps it eligible AUTOMATICALLY, and a fully-racked lot's CCT is null → dropped. The sim becomes the single, correct predicate. Q2 was only ever a cheap pre-trim; with the volume-aware sim it's redundant AND incorrect for partials. **Remove it.**
   - **(ii) If keeping a cheap pre-filter:** replace Q2 with `NOT EXISTS(bd_racking_v2 rk WHERE …lot… AND rk.is_tombstoned=0 AND rk.transfer_interrupted=0)` — i.e. eligible unless a NON-interrupted (complete) rack exists. This still mis-handles "interrupted twice cumulatively ≥ brewed volume" (sim catches it, EXISTS can't) → exactly why (i) is cleaner. The cumulative-racked-vol≥source-brewed alternative is precisely what the sim already computes by decrement — don't re-implement it in SQL (2nd divergent occupancy model = forbidden).
   - **NET: the predicate is "source CCT is non-null in TankSimulator after replaying all events incl. this interrupted rack's partial decrement." Make the sim volume-aware; let simFilterCct be the gate; delete Q2 (or downgrade it to transfer_interrupted=0).** beer_lost on a full rack → CCT null → ineligible (correct; the beer is gone). beer_lost on a partial → remainder stays → eligible (correct; un-lost remainder is still transferable).

**3. Loss accounting now-vs-later — RECORD-ONLY THIS BUILD. Hard boundary.** beer_lost = WIP shrinkage, real, but the COGS/COP/WIP pipeline currently attributes WIP via TankSimulator occupancy × brew-cost/HL (the FG-valuation-uses-prev-month-WIP chain). **This build: record `beer_lost`/`transfer_interrupted`/reason on the row + let the volume-aware sim NOT deposit the lost volume (so WIP/occupancy already self-corrects downstream — the lost HL simply never appears in any tank).** Do NOT this build: write an inv_consumption row, a dedicated loss-event table, or a COGS flag. Rationale: (a) the sim NOT-depositing already removes the phantom HL from WIP valuation (the loss is reflected as absence, the event-sourced way — no double-entry needed); (a COGS *loss line* would be a NEW accounting fact requiring its own model + GL decision = scope creep + a guess). (b) A loss-event ledger is a real model decision (where does destroyed-beer cost land — variance? waste GL? write-off?) that needs the operator + a COGS ruling, not a checkbox's side-effect. **Boundary: this build makes the loss VISIBLE (flag+reason on the row, KPI-countable) and PHYSICALLY CORRECT (sim doesn't deposit it). A loss→COGS ledger is a SEPARATE future build (RULE-1 backlog), gated on the operator defining the accounting treatment. Refuse-don't-guess: never invent the GL/treatment of a destroyed-beer cost from a checkbox.**

**4. Source-tank total volume — DERIVE from sim at submit, NO new operator input. CONFIRMED.** At POST, after building `$simState`, read `$srcVol = $simState['cct'][$selectedCctNum]['volume_hl']` for the lot's source CCT (the form already has the CCT number from the selected card + already runs the sim). **Full-vs-partial determination (deterministic): `isFullEmpty = (racked_vol_hl >= $srcVol − EPSILON)` where EPSILON ≈ RACKING_LOSS_HL or a small tolerance (e.g. 0.5 HL) to absorb measurement noise.** racked_vol ≥ source total → full loss/empty; racked_vol < source total → partial, remainder stays. This is computed in the SIM (point 1's decrement-then-threshold), NOT re-derived separately in the form — the form needn't store srcVol; the sim owns the arithmetic on replay.
