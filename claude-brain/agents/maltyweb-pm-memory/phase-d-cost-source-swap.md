# Phase D — COGS/COP cost-source swap (BSF → MySQL v_mi_cost)

> The arc that wires COGS/COP to read MI cost from the MySQL canonical store (`v_mi_cost`) instead of legacy BSF `MI.price`. Soft rollout, cost-neutral on the gap set, FX-careful on the consumers. Companion: billing-format-and-wac-unit-model.md (the WAC view + catalog-hygiene that feed this). Verify read-only before recording.

## GOAL / shape
Replace the BSF `MasterIngredients.price` cost source used by the JS COGS/COP pipeline with the MySQL canonical cost. The bridge = port `lib/mi.js loadMI` to read MySQL and carry a `v_mi_cost`-derived cost; all JS consumers stay transparent. Retire the BSF writeback (`compute-weighted-prices --write-master`) at the end.

## COST RESOLVER MODEL (decided + live)
`v_mi_cost` is a THIN per-MI cost resolver. `cost_basis` precedence:
1. **wac** — `v_mi_wac.wac_chf` (already CHF). The 96 delivery-backed MIs.
2. **catalog** — `ref_mi.price × ref_fx` where `price IS NOT NULL` (incl. 0 = a valid free/bundled cost). 141 MIs.
3. **no_basis** — `price IS NULL` → `cost_chf NULL`, **never 0**. 130 MIs.

Live counts (2026-05-27, mig 178): 367 MIs = wac 96 / catalog 141 / no_basis 130.

**RESOLVER BUG caught + fixed before commit (durable lesson):** the original predicate was `COALESCE(price,0) > 0`, which mis-classified stored `price = 0` (free samples like ROTTENBURGER/TANGO) as `no_basis`. Fixed to `price IS NOT NULL` → `price = 0` correctly yields catalog cost 0. **RULE: `price IS NULL` is "no cost basis"; `price = 0` is "a real zero cost." Never collapse them with COALESCE-to-0.** 178 was re-applied (idempotent) after the fix.

## ref_fx (mig 178) — the FX table
`ref_fx(currency PK, rate_to_chf)`: CHF = 1.0, EUR = 0.945. **Replaces the hardcoded 0.945** scattered in the JS pipeline. Operator-editable. This is the single home for the EUR→CHF rate going forward — the "never hardcode" rule applied to FX.

## v_mi_cost_gap (mig 178) — the diagnostic, NOT the review queue
Surfaces `no_basis` MIs (`price IS NULL`) that are ACTUALLY in use: in an active SKU BOM **OR** held in RM stock (`inv_rm_stocktake` latest period). Dormant/unused no_basis MIs never surface — by design. This is a diagnostic view, not a `doc_review_queue` writer.

## ⚠️ STEP-3 PREREQUISITE — input_unit→pricing_unit CONVERSION GAP in inventory.js (LOAD-BEARING, surfaced 2026-05-27 by the Step-3 read-only diff harness)
The Step-3 harness showed a +CHF 564k / +169% RM inventory swing. Decomposed: it is NOT a real revaluation — it is a **unit-conversion-application gap** in the inventory VALUATION layer. PM-RULED + PM-VERIFIED LIVE 2026-05-27.

**The defect (verified by reading the files live):**
- `lib/inventory.js` `parseRMRows` L96-97 values RM as `priceCHF = currency==='EUR' ? price*EUR_CHF : price; valueCHF = qty × priceCHF` — **ZERO input_unit→pricing_unit conversion.** It multiplies a gram qty by a per-kg price for every g/kg MI.

**CONVERSION SOURCE — RULED (c)-ALREADY-SHIPPED: read `ref_mi.conversion_factor` from MySQL. The finding's "ref_mi has NO conversion column" premise was a FALSE NEGATIVE** (the column is named `conversion_factor`, not `conversion`). - REJECT (b) derive from unit strings ÷1000 — mis-handles ml→mass (assumes density=1; the lone ml→kg PROC_AMS needs density), and re-derives a fact the canonical table already owns = parallel-store anti-pattern.

**SEQUENCING — RULED: SEPARATE PREREQUISITE COMMIT, validated on its own, BEFORE the cost swap.** The conversion fix corrects a defect wrong TODAY regardless of the swap, so it gets its own RM-valuation before/after diff. This makes the +564k decomposable: conversion-fix delta (hops/minerals collapse to sane; yeast unchanged) vs the real WAC-revaluation delta (96 WAC MIs only, on a now-correct unit foundation). ## ⚠️ STEP-3 RISK FINDING — the FX double-conversion landmine (load-bearing; SURFACE WIDER THAN ORIGINALLY RECORDED)
**CORRECTION (2026-05-27, file-verified against the maltytask repo):** the FX self-conversion surface is NOT just 2 files. The MI-entry `.currency` field is read for EUR→CHF self-conversion in **FIVE** places, mostly via a SHARED helper:
- **`lib/pricing.js::toCHF(amount, currency)`** (L78-79) — `if (currency === 'EUR') return amount * EUR_CHF`. The shared self-converter.
- Called by **`scripts/build-cogs.js`** (L66, L254 WIP), **`scripts/build-cogs-dashboard.js`** (L229), **`scripts/build-month-closure.js`** (L233).
- **`lib/inventory.js`** L96 inlines the same logic (`currency === 'EUR' ? price*EUR_CHF : price`) — RM valuation.
- **`lib/pricing.js`** L61 reads `mi.currency === 'EUR'` to set `eurBase` (EUR-exposure figure).
- **`scripts/build-sku-bom.js`** (~L219-615, ~25 sites) uses `priceInfo.price` **RAW — NO FX applied**.

So `v_mi_cost.cost_chf` (already CHF) **cannot be dropped into `MI.price` blindly**: it would double-convert in every `toCHF()`/inventory consumer AND surface/fix latent EUR-unconverted lines in build-sku-bom (which were silently under-costing EUR MIs).

**THE CLEAN LEVER (decided):** the overlay sets `price = cost_chf` AND **`currency = 'CHF'`**. Setting currency to CHF makes the EUR branch a no-op in ALL five self-converters at once (toCHF passes through, inventory passes through), and build-sku-bom's raw multiply of an already-CHF value is then correct. One lever neutralizes the whole surface.

**NO non-price consumer of MI.currency exists** — grepped every MI-entry `.currency` read; all are EUR→CHF math or `eurBase`. The other `.currency` reads in the pipeline are on DELIVERY/INVOICE rows (reconcile, ingest-documents, build-cogs L153 delivery rate, rematch, add-ingredient write-path) — NOT the MI entry; the overlay doesn't touch them. So flipping MI.currency→CHF is safe for cost math and corrupts nothing else.

**CAVEAT to flag in the diff:** `lib/pricing.js` L61 `eurBase = (currency==='EUR' ? price : 0)` collapses to 0 for every MI under the overlay. If any LIVE report surfaces EUR-exposure from `eurBase`, that silently zeroes — a reporting-semantics change (NOT a COGS-number change). Verify whether eurBase is consumed; call it out if so.

**HARNESS must diff ALL FOUR cost-consuming report builders** (build-cogs-report + build-cogs + build-cogs-dashboard + build-month-closure), not just build-cogs-report — the toCHF neutralization touches all of them. Expect deltas only on the 96 WAC MIs + a CORRECT EUR-fix delta in build-sku-bom (latent under-costing now fixed — flag as correct, not breakage). **This WILL move COGS → mandatory before/after diff + sign-off.**
