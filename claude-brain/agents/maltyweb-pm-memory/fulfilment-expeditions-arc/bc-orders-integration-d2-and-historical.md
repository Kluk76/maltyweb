## 🔑 D2 ECHO-FIELD RULING — credit-memo write-back (PM ruling 2026-06-15, BC-probe-backed)
**Context:** D2 (credit-memo write spine, disarmed, nothing committed) surfaced a surface-split conflict. The avoir (credit memo) entity is NOT on the OData v4 web-service surface D1 uses (`/ODataV4/.../SalesCreditMemo*` → 404). It lives ONLY on **BC API v2.0** (`/api/v2.0/companies(<uuid>)/salesCreditMemos`), where `Your_Reference` does NOT exist — the only filterable echo field is **`externalDocumentNumber`** (= `External_Document_No`), the field RESERVED in the D1 ruling for SalesOrders.

**LIVE BC PROBE (2026-06-15, read-only, NEBULEUSE company id `e2b691c6-8393-ed11-bff5-002248f307ee`):** 3,800 credit memos total; **155 (4%) have `externalDocumentNumber` populated**; sample values = free-text human refs (`RETOUR DOWTOWN LAUSANNE`, `COMMANDE 127596`, `CMD: N° 4031§`, `ERREUR QUANTITE`, bare invoice numbers). **ZERO start with `mt:`** → namespace collision-free. `$filter=externalDocumentNumber eq 'mt:cm:<n>'` confirmed OData-filterable → GET-before-create idempotency works.

**RULING (option 3 + 2a hybrid):**
1. `externalDocumentNumber='mt:cm:<order_id>'` is **ACCEPTABLE as the echo for CREDIT MEMOS** (NOT for sales orders — the D1 reservation stands unchanged on the OData SalesOrder surface). Three reasons: (a) the D1 reservation was framed around `External_Document_No` as KOUROS'S split-order key — that key lives on the SALES ORDER flow, not on credit memos; on the CM entity the field is a free-text human ref, sparsely used (4%), and our namespaced `mt:cm:` prefix is provably disjoint from every existing value AND from the order namespace `mt:` (bc_echo.py enforces the disjointness in code: `bc_echo_parse` returns None for `mt:cm:…`, `bc_cm_echo_parse` returns None for `mt:…`). (b) it is the ONLY filterable echo field on the v2.0 CM entity, so GET-before-create idempotency has no alternative anchor on the BC side. (c) namespacing makes accidental collision with a human-typed order ref structurally impossible.
2. **BUT the echo field is the SECONDARY idempotency guard, not the primary.** Primary = the `ord_bc_credit_memos` tracking table (mig 366): a local row keyed on the source order means "we already issued an avoir for this." The BC-side GET-before-create on `externalDocumentNumber eq 'mt:cm:<id>'` is the cross-process safety net (catches a tracking-table/BC divergence — e.g. tracking write failed after BC create). Both must agree before a CREATE; either one matching = skip-and-rekey, never double-issue. (This is option-2a's tracking-table idempotency layered UNDER option-3's namespaced echo — belt and suspenders, because a credit memo is financially irreversible.)
3. **SURFACE-SPLIT IS FINE — no reader/echo-loop problem.** The reader (`ingest_bc_sales_orders.py`) pulls ORDERS only and never pulls credit memos back; nothing in maltytask consumes the avoir on a later pull. So the CM "echo" exists SOLELY for GET-before-create idempotency, NOT a read-loop — which is exactly why the field choice is low-stakes (no downstream parser keys on it). The re-invoice (document #2) continues through D1's OData SalesOrder path with `Your_Reference='mt:<id>'`; the avoir (document #1) goes through API v2.0 with `externalDocumentNumber='mt:cm:<id>'`. Two surfaces, two disjoint namespaces, one tracking table — coherent. The ONLY thing that must hold: `bc_cm_echo_parse` must never be fed to the order reader and vice-versa (already structurally guaranteed by bc_echo.py).

**CONFIRMED UNCHANGED from prior D2 scope:** CREATE-only, NEVER POST/post (avoir + re-invoice stay in draft/unposted). Disarm = dry-run default + two-flag live (`--apply` AND `--i-have-kouros-go`) + NO cron. The COGS verification Opus owes BEFORE any write (the DGDF-wrong-vs-DGDB-right line depletes correctly) is STILL REQUIRED — run it as the rolled-back-txn probe, same discipline as D1's id=365 no-double-deplete check.
