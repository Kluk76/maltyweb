**THE 7 RULINGS:**
1. **SOURCE = standard `salesOrders` API as set-driver** (29 live, doc-state-scoped, drops posted/shipped automatically; the custom `SalesOrder` web service returns stale 2020 orders = filtering liability). **Custom web service = TARGETED by-No lookup ONLY** to enrich each in-set order with `ShpfyOrderNo` + `Shipping_Agent_Code` IF the standard header lacks them. **PROBE FIRST** whether standard `salesOrders` exposes `ShpfyOrderNo` (some BC tenants surface it as a custom API field) — if yes, skip the hybrid entirely. Reuse OData idioms verbatim: S2S OAuth2 client-creds `config/bc.env`, OR-chained `$filter` at **50 ids/batch (BC rejects the `in` operator, HTTP 501)**, **nextLink paging, NO `$top`** ($top suppresses @odata.nextLink → silent truncation). `Shipping_Agent_Code` → `ref_transporters`.
2. **SHOPIFY DOUBLE-COUNT EXCLUSION = `ShpfyOrderNo` non-empty → EXCLUDE.** A BC sales order carrying ShpfyOrderNo IS an eshop sale already in `inv_sales_orders` (channel='eshop', deplets fg-stock Step 5). Pulling it into ord_orders double-counts on the board AND in fg-stock. Belt-braces: ALSO exclude system customers `1080` (eshop privé) + `3822` (taproom) — a B2B open order should never be for them. Net filter = `ShpfyOrderNo empty AND Sell_to_Customer_No NOT IN (1080,3822)`. No other channel leak today (Lightspeed has no feed).
3. **CUTOVER = sequenced cut, NO dual-live window.** Stop running `import_weeklyorders.php`. Add ENUM val: `ALTER TABLE ord_orders MODIFY source ENUM('web','email','import','bc') NOT NULL DEFAULT 'web';`. BC orders key `source='bc'`, `source_ref='bc:<BC OrderNo>'` (e.g. `bc:ORD201126`) — `source_ref` already UNIQUE → idempotent upsert. **Existing web/import rows AGE OUT — do NOT migrate/reconcile** (50 import + 16 web = real operator-worked orders; shipped ones won't reappear as open BC orders since BC closes on full shipment). The ~39 genuinely-open import/web rows MAY collide with the open BC set → 1st dry-run **match-report flags (customer, requested_date, lines) collisions** → operator decides per-row (cancel legacy vs skip BC pull). NEVER auto-merge. After one operator backlog-clear pass, BC is sole feed.
4. **🔴 CRUX — STATUS PRESERVATION on recurring pull.** Single rule: *the BC pull is a HEADER/LINE refresher gated on `status IN('entered','confirmed')`; status + status_events are OPERATOR-OWNED and NEVER written by the pull.* On INSERT: create order (status='entered') + lines + ONE 'entered' event. On re-pull of existing (matched by source_ref): UPDATE only `customer_id_fk`/`requested_date`/`transporter_id_fk`/`comment`/line qtys+SKUs — DO NOT write `status`, DO NOT append a status event. Freeze lines once status >= 'picked' (a BC line edit must not silently change what's being picked → flag divergence to review). `Completely_Shipped=true` / order leaves the open BC set = a **reconciliation SIGNAL surfaced on the board** ("BC reports shipped"), NOT an auto-advance/cancel — auto-advancing would (a) write unauthorized status and (b) fire fg-stock depletion on a pull not an operator action (wrong lane). Order leaving the open set must NOT delete/cancel the local row (it's mid-fulfilment locally).
5. **SKU RESOLUTION = `(Item_No, Unit_of_Measure_Code)` → ref_skus.** Item No ALONE is insufficient (same beer item ships in multiple formats; UoM code e.g. 24B selects the format/SKU). Ledger connector resolved on bare Item_No only because the ledger export is already SKU-grain; an ORDER line is item+UoM. Build/confirm a `(bc_item, bc_uom) → sku_id` resolver, reuse `ref_sku_aliases` where it already encodes the combination. **REFUSE-DON'T-NULL:** an unresolved line does NOT create a `sku_id_fk=NULL` line (RESTRICT FK forbids it anyway; a NULL line silently under-picks + under-deplets). Emit a self-sufficient review-CSV row (order No + raw item + UoM + qty, same pattern as the ledger connector's unresolved-SKU CSV) and hold the order out of apply OR apply flagged `review_status='pending'` with the bad line surfaced — operator-decided, NEVER guess item+UoM→SKU from a pattern.
6. **REFRESH = VPS cron `*/15`** (match `maltytask-shopify-ingest` cadence; open set tiny ~29 + operationally hot — logistics wants new orders visible fast; do NOT use the ledger's 4h). User maltytask, `--apply`, log to `/var/log/maltytask/`. NOT local. Webhook = Phase-2-era nicety, don't build now.
7. **MIGRATION = ONE:** the `source` ENUM extension above (MySQL-8, no IF NOT EXISTS; ENUM extension is INSTANT). `source_ref` UNIQUE already exists. **No new table → no schema_meta row.** Possibly a small `bc_uom→format` lookup if not derivable via aliases — prefer reusing `ref_sku_aliases` over a new table; confirm in build. Pre-flight: `migrate.php --status` (head was 345 at last check — RE-VERIFY, parallel sessions lead), next free number, no bare SELECT.

**RULING — PHASE 2 WRITE-BACK ARCHITECTURE (decision 3), the hard parts:**
  - **(a) ECHO-LOOP:** maltytask creates order in BC → `*/15` reader pulls it back → MUST NOT duplicate. Mechanism = maltytask writes a maltytask-owned external ref into a BC field the reader keys on; reader's source_ref upsert (`bc:<No>`) is already idempotent on BC OrderNo, but the maltytask-born order has a LOCAL id BEFORE BC assigns a No. **Resolve by: maltytask-born order gets a provisional local source_ref (`mt:<localid>`); on successful BC create, store the returned BC No and REKEY source_ref → `bc:<No>` in the SAME write transaction; the reader then matches it as existing on next pull (UPDATE not INSERT).** Do NOT rely on External_Document_No for echo identity (see b).
  - **(b) 🔴 External_Document_No DUAL-USE COLLISION — FLAG:** Kouros ALREADY uses BC `External_Document_No` to distinguish SPLIT orders (same customer, different location). Writing maltytask refs into it would COLLIDE with that operational use. → Phase 2 must NOT hijack External_Document_No for the maltytask echo ref. Use a DISTINCT BC field (a custom field e.g. `MaltytaskRef`, or `Your_Reference`, or the Shopify-style `ShpfyOrderNo` pattern of a dedicated custom column) — confirm with Kouros which BC field is free. This is a hard prerequisite for D's create path.
  - **(c) IDEMPOTENCY + PARTIAL FAILURE:** every write carries the maltytask local id as the dedup key (the provisional ref); a create that times out but actually landed in BC must not double-create on retry → before create, GET BC by the maltytask ref; create only if absent. Wrap create+rekey in a local transaction; if BC create succeeds but local rekey fails, a reconcile pass (the reader) recovers by matching the BC ref back to the local provisional row.
  - **(d) CREDIT-MEMO (Pepitium):** create avoir (`salesCreditMemos`) for the wrong locked invoice + corrected re-invoice. Start with CREATE only (per Kouros: CREATE orders + credit memos; POST stays human in BC). The B-flag (`correction_compta_requise`) is the queue that feeds D's credit-memo generation.
  - **(e) BC API CHOICE:** standard `salesOrders` / `salesCreditMemos` POST (the standard Business Central v2.0 API) for CREATE — cleaner than OData web services for writes, supports the standard entity shapes. Reuse the S2S OAuth2 client-creds from `config/bc.env`. (The read side uses OData `SalesOrder`/`SalesOrderSalesLines` web services for richer fields; writes can use the standard API independently.) BC write permission = Kouros's BC-side permission-set change (SUPER role, Entra has API.ReadWrite.All) — he does it himself; we build assuming it lands. **Start with CREATE, NEVER POST (no posting/invoicing from maltytask initially — keep that human in BC).**
