# UI / Rendering Coder (UI)

This skill is the **rendering layer** discipline for the maltyweb operator app — the half of the work that decides whether something actually shows up correctly in the operator's browser. It encodes a real catalog of rendering bugs (each cost a debugging session) plus the maltyweb design system and the `frontend-design` aesthetic principles.

**Read once at task start. Re-consult the Rendering-bug catalog and the "popup/preview doesn't work" playbook before declaring a UI change done.**

## Project manager — align with `maltyweb-pm`
A standing **`maltyweb-pm`** project-manager agent (`~/.claude/agents/maltyweb-pm.md`; knowledge base `~/.claude/agents/maltyweb-pm-memory.md`) is the keeper of the UI build sequence + project state (which pages are live vs design mocks, the current phase, the light-kraft recolour roadmap) and the Le Zeppelin information architecture (the families: Salle des Machines / Salle de contrôle / Le Cockpit / Données générales — the source-of-original-truth from which everything derives). For any maltyweb UI build: align with it — which page hosts the work, where it sits in the IA, the design-system + palette, and capability-gating (only render formats/elements the commissioned equipment supports). After a page ships or the build sequence advances, **its memory must be updated** by whoever landed the change.

## When to use this skill

- Building or integrating an operator-facing **page / module / fiche / dashboard / modal / chart** in maltyweb (`/home/kluk/projects/maltyweb`, deployed to `/var/www/maltytask`).

## maltyweb render pipeline (how a page actually works)

```
ref_* / doc_* / inv_* (MySQL, canonical)
        │  PDO query in public/modules/<page>.php   (sql skill territory)
        ▼
  PHP assembles $rows  →  echo 'window.SUPPLIERS = ' . json_encode($rows,
        JSON_UNESCAPED_UNICODE | JSON_HEX_TAG | JSON_HEX_AMP) . ';'
        ▼  (server-rendered <script> blob — NOT an AJAX call)
  public/js/<page>.js  reads window.SUPPLIERS, renders list / fiche / modal
        ▼
  public/css/app.css   styles it, all scoped under a <body> class
```

Key facts:
- **Data is injected, not fetched.** Most pages embed `window.X = {...}` server-side; the JS hydrates from it. There's no API round-trip for the initial render. (Media endpoints like `/api/document-preview-png.php` ARE fetched.)
- **CSS/JS are external and cache-busted.** PHP links them with `?v=<?= filemtime(...) ?>`. A deploy changes mtime → the `?v=` changes → browsers fetch fresh. So a *normal* reload picks up new JS/CSS after deploy (no hard-reload needed), PROVIDED the HTML page itself is re-fetched.
- **A service worker (`public/sw.js`) controls scope `/`.** It is a pure pass-through (`respondWith(fetch(event.request))`) with `skipWaiting`+`clients.claim` — no offline caching. It is NOT a caching layer to fight, but it IS in the request path; keep it in mind when reasoning about what hits the server.
- **CSP is Report-Only** (`Content-Security-Policy-Report-Only`, `img-src 'self' data:`). It does not block — but if it ever flips to enforcing, same-origin + `data:` images are the only allowed image sources.

## Design-system consistency rules

These apply to every page and component — check before declaring any CSS work done:

- **Tinted shadows only.** Never raw `rgba(0,0,0,…)` on the warm kraft theme. Tint toward `--oak`
  or a warm-dark equivalent (e.g. `rgba(30,18,5,0.12)`). A black shadow on kraft paper reads as
  an artifact from a different design system.
- **Corner-radius lock.** One documented radius tier per element class — interactive chips get
  their radius, cards get theirs, modals get theirs — applied consistently everywhere. Mixing
  radii on same-role elements reads as unfinished.
- **Theme-family lock.** Section backgrounds stay within `--bg`/`--bg-elev`/`--bg-side`. Never
  an alien `#111` block mid-page on a kraft-paper layout — it reads as a copy-paste from a dark
  theme. If contrast is needed, step to a darker shade of the same family.
- **No pure `#000` anywhere.** Off-ink (`--ink`) or tinted dark only.
- **Accents < 80% saturation.** Oversaturated accents compete with data and read as decorative
  rather than semantic. The `--hop`/`--ember`/`--oak` tokens are already calibrated — don't
  invent new hex.

## maltyweb design system (match it exactly — see [[maltyweb-frontend-aesthetic]])

- **Colour palette — light kraft/manila theme (dark ink on paper), the site-wide house standard.** It is defined by a single global `:root` token block at the top of `app.css` that every body class inherits — there is NO per-body-class palette re-declaration. **Recolour & new-CSS rule: COLOURS ONLY, via tokens — NEVER per-page hardcoded hex** (violates single-source + CSS-in-`/public/css`). Structure/sketches/lines/fonts/layout are theme-independent; only token *values* define the look, so a consumer using `var(--bg)`/`var(--ink)`/`var(--hop)` adopts the theme for free. Any mid-tone token used as TEXT must pass WCAG AA on paper `#f1e8d4` **and** against the actual surface it sits on (use the darker `-deep` variant on elevated `--bg-elev` cards). **Don't invent hex, and don't invent tokens** (`--paper*`/`--copper*`/`--ice*` → map to existing ones, e.g. `--copper → --ember`). **Before any recolour or new-page CSS pass, get the current token values, surface-contrast caveats, and SVG-tank treatment from `maltyweb-pm`** — it holds the resolved as-built.
- **All page styles scope under a `<body>` class** (`body.home`, `body.approv-fournisseurs`, …). Unscoped rules either never apply or leak across pages. New page variation → new body class, reuse the one `app.css`.
- **Type stack** (loaded globally): `Fraunces` (display serif; `font-variation-settings:"opsz" 96|144`, italics for beer names), `DM Sans` (body), `JetBrains Mono` (labels/readouts, wide letter-spacing on uppercase).
- **Canonical tokens** (consume via `var(--…)`, never raw hex): surfaces `--bg`/`--bg-elev`/`--bg-side`, ink `--ink`/`--ink-soft`/`--ink-mute`/`--ink-label` (faint TEXT/overlines/labels — #8a7250)/`--ink-faint` (divider/border/tint ONLY — #c8b48a, never a text color), accents `--hop`/`--ember`/`--oak`/`--cold`/`--bbt`, stainless `--steel*`. They are declared once in the global `:root`; the resolved value table + per-surface AA-contrast notes are kept by `maltyweb-pm`.
- **Reusable SVG tank library** at `app/svg-tanks.php` (`svg_cct()`, `svg_bbt()`). CALL it from PHP — never re-inline a tank. On light-ish backgrounds wrap in a dark display-case card.
- **No DB nomenclature in the UI** (see [[ui-no-db-nomenclature]]): render full human terms (`fmt_suffix='V'` → "Cuve de service", `run_type='bot'` → "Run bouteille"). EXCEPTION: SKU codes (`ZEPF`, `EMB4`) stay compact in the production surface, full label in SKU cards. Keep a label-map layer; reverse-map on submit.
- **No hardcoded operator-configurable data in any rendered surface — read it from `system_settings`/`ref_sites`** (the Zeppelin "Données générales" page). Page headers, footers, email templates, report titles, "about" blurbs must pull the brewery name / address / site city·country / date·time format from the DB, never bake a literal string. Real incident (2026-06-04): an invite-email footer AND the set-password page hardcoded `La Nébuleuse · Brasserie artisanale · Neuchâtel` / `Lausanne · CH` — invented; the "Neuchâtel" was lifted from a greyed-out `placeholder=` attribute on the settings city input — while the truth was `system_settings.brewery_name='La Nébuleuse'` + the `ref_sites` production row = **Renens, CH**. Use a `brewery_identity()` helper (`app/settings-helpers.php`) with a try/catch literal fallback so a settings read never fatals the render. A `placeholder=` is example text for humans, NEVER a data source. See [[feedback_system_settings_central_no_hardcode]].
- **CSS in `public/css/*.css`, JS in `public/js/*.js` — never inline `<style>`/`<script>` in PHP** (see [[css-separated-from-php]]). Scope new component CSS with the body class + a component prefix (e.g. `body.approv-fournisseurs .af-inv-*`).

## Design-system architecture — the three-layer reskin contract

A robust design system is **three layers, each built only from the one below; you reach DOWN a layer only when no higher one fits — never skip straight to bespoke CSS.** Get this right and one token change reskins the entire app; get it wrong and the look fragments into per-page hex. maltyweb already half-lives this — the discipline is to make it explicit and never bypass it. (Mined 2026-06-19 from a Tailwind/CVA design-system card; the *strategy* is stack-independent — the maltyweb mapping is below, not the Tailwind/React specifics.)

- **Layer 1 — Semantic tokens (one source).** A small fixed set of CSS custom properties in ONE global `:root` block, named by *role* not by hue (`--bg`/`--ink`/`--hop`, never `--brown`/`--green`), each with its systematic variants (maltyweb: `-deep`/`-soft`/`-mute`/`-label`/`-faint`; the reference card's app used DEFAULT/light/dark/ring/text per token). **Change the brand colour in one line.** This is the load-bearing layer — never declare a per-page palette, never hardcode hex (the COLOURS-ONLY rule above). A *new* token is a system-wide decision: clear it with `maltyweb-pm`, don't invent `--paper*`/`--copper*`/`--ice*` (remap to existing — `--copper → --ember`).
- **Layer 2 — Composed, token-backed classes.** A library of named component classes (button, table, tab-strip, filter, form-row, card, chip), each built ONLY from Layer-1 tokens, scoped under the body class. **Reach for an existing composed class BEFORE writing custom one-off CSS.** The standing temptation in a PHP module is to hand-roll a `<div style="…">` or a fresh `.xyz-thing` — that bypasses the system and drifts. If a recurring pattern has no class yet, add ONE composed class (token-backed, body-scoped) and reuse it; never copy-paste declarations across pages.
- **Layer 3 — Reusable render primitives (call, never re-inline).** A roster of shared render helpers — maltyweb's `svg_cct()`/`svg_bbt()` (`app/svg-tanks.php`), the topbar / PageHeader partials, `kpi_dispatch()` tiles, `brewery_identity()`. **Call the helper; never re-inline its markup** (the SVG-tank rule, generalised). A primitive carries its own correct paint order, escaping, and token wiring — re-inlining forks all three and silently drifts. When you build a new reusable surface (a card, an empty-state, a status pill, a KPI tile), make it ONE callable helper with a single definition — the maltyweb analogue of a barrel-exported component.
  - **The config-driven shared partial — the cross-page form of Layer 3.** When the SAME embeddable surface must appear on several different pages, do NOT copy its markup into each page. Build it once as a partial under `app/partials/` (or `public/modules/partials/`) that renders itself from a `$config` array the *caller* sets before `require`, and that does ZERO DB queries of its own — the caller supplies the options/data. The host page's only job is: set the config (from its own PDO query), `require` the partial, and link the partial's dedicated `/css/<name>.css` + `<script defer>` `/js/<name>.js`. The shared JS is an IIFE driven by `data-*` attributes the partial renders (`data-type`, `data-endpoint`), so one script file serves every host. Adding the surface to a new page is then four lines, not a fork. Two live instances prove the pattern: the **CIP module** (`app/partials/cip-section.php` driven by `$cipConfig`, live on racking/brewing/packaging) and the **Lookup Panel** (`public/modules/partials/lookup-panel.php` driven by `$lookupConfig`, with a per-page read-only `/api/<page>-lookup.php` JSON endpoint). The win is the same as a single callable helper — one definition, one place to fix a bug, no per-page drift — extended across page boundaries.
  - **Read-only GET JSON endpoints that back such a panel: gate, but no CSRF.** A `/api/*.php` endpoint that only SELECTs and returns JSON still needs `require_page_access('<page>')` — the gate is what stops it leaking data the operator's role shouldn't see — but it does NOT need `csrf_verify` (CSRF protects *state changes*; a read has nothing to forge). Validate any date param with `checkdate()`, not `strtotime()` — `strtotime` silently accepts overflow dates ("2026-02-31") and returns a plausible-but-wrong day, which then drives a wrong-data lookup with no error. In the JS renderer, a NULL numeric from the DB (e.g. an un-computed `vendable_hl`) must render as `—`, never coerced to `0` — `0` reads as a real measured zero and hides the "not yet known" state.

**Page-layout nesting is mandatory, not optional.** Every page obeys the same shell composition top-to-bottom: role-gated app shell → width-capped page shell → page header (title, actions, back-link, breadcrumbs) → content → for hub pages: tab-strip → panel → *active* tab-panel. In maltyweb this is `body.<page>` under the global shell + topbar + `.home .main` (width cap / fit-guard) + the `.tab-strip` / `.panel` / `.panel--active` pattern — **drive panel visibility from the state class, never the `[hidden]` attribute** (see the stacked-panels trap above). Don't invent a bespoke outer scaffold per page: slot into the nesting so role-gating, max-width, keepalive, and the back-link behave identically everywhere.

## Design workflow: preview-first (see [[design-preview-workflow-maltyweb]])

For any *new* component/page/modal, do NOT integrate into the live PHP module on the first pass:
1. Build a self-contained mockup at `public/_design/<name>.html` (inline style/script, CDN fonts, hardcoded mock data) — match the dark theme exactly.
2. `cd /home/kluk/projects/maltyweb && bin/deploy.sh --apply` (only `public/ app/ db/ scripts/ data/` are rsynced; `public/_design/*.html` is web-accessible, no auth).
3. Share `https://app.maltytask.ch/_design/<name>.html` for visual approval; iterate as static HTML (cheap).
4. Only then integrate: port CSS into `app.css` (prefix with the body class, remap tokens), wire data via SQL in the PHP module, port interaction JS into `public/js/<name>.js` (mock data → `window.X` embed). Use frontend-design thinking in step 1; use this skill's catalog in step 4.

**When upgrading an existing module page**, apply changes in this order for maximum impact per
unit of risk: ① font/type adjustments → ② color token cleanup → ③ hover/active states →
④ layout/spacing → ⑤ replace generic components → ⑥ add loading/empty/error states →
⑦ typographic polish. Each step is independently shippable.

## Rendering-bug catalog (each was a real incident — check before declaring done)

- **Lazy image while hidden = permanent deadlock.** An `<img loading="lazy">` that starts `display:none` (e.g. shown only after `onload`) NEVER loads: a hidden, lazy image never enters the viewport, so the browser defers the fetch forever — **zero network requests**, `onload`/`onerror` never fire, spinner sits on "Chargement…". Symptom that fingerprints it: the direct image URL works in a tab, but the in-page `<img>` is stuck and the server log shows no hit. **Fix: drop `loading="lazy"`** for images revealed on demand (modals, tabs). Lazy is only for long, naturally-scrolling lists of visible images. (2026-05-25, invoice-preview modal.)
- **`<dialog>` `display` trap.** Applying `display:<anything-but-none>` unconditionally to a `<dialog>` makes it PERMANENTLY visible (backdrop and all) on page load. Scope layout to `[open]`: base reset (`border:0;padding:0;background:transparent`) + `dialog.x[open]{ position:fixed; inset:0; display:grid; … }`. Verify by toggling `open` in DevTools. (see [[dialog-modal-gotchas]])
- **`<dialog>` top-layer trap.** An open modal `<dialog>` renders in the browser top layer; tooltips/popovers/portal targets appended to `document.body` render BEHIND it and look broken. Mount helper elements INSIDE the dialog (`position:fixed` still uses the viewport there).
- **A `.class { display:… }` rule overrides the HTML `[hidden]` attribute → tabbed/toggled panels all show at once.** The `hidden` attribute is just UA `display:none`, which ANY author rule with a `display` value beats on specificity (`.fin-panel{display:flex}` 0,1,0 ≥ `[hidden]` UA). Symptom: every tab/accordion panel renders stacked simultaneously, content is correct, and clicking tabs "does nothing" (the JS toggles `hidden`/`--active` but CSS keeps them all `display:flex`) — reads as "the sections don't work." Fingerprint: page has all the data, no console error, but multiple panels visible top-to-bottom in a full-page screenshot. **Fix: drive visibility from the state CLASS, not `[hidden]`** — `.panel{display:none}` + `.panel--active{display:flex}` (the tab JS already toggles `--active`; the default-open panel carries it in the server-rendered HTML). Don't rely on the `hidden` attribute when the element also has a `display` class. This is the exact mirror of the `<dialog> display` trap above. (2026-06-10, financier.php four panels stacked; fix `f1ff3a1`. Caught only by a manager-auth full-page screenshot — `php -l` + data-layer checks pass a stacked page; this is why a real render screenshot is the load-bearing UI gate.)
- **Server-rendered initial visibility of a JS-toggled field group must match the DEFAULT-selected control — else no-JS / pre-JS-paint users see the wrong fields.** A form with a `<select>` whose value JS uses to show/hide dependent field groups (an `initWortProcessToggle`-style toggle) has TWO sources of truth for "which group shows first": the server-rendered `hidden` attributes, and the select's default-selected option. If they disagree — the select defaults to the first `<option>` (`brewing`, no explicit `selected`) but the server renders the *brewing* group `hidden` and the *non-brewing* group visible — then before JS runs (or with JS disabled / a slow tablet) the form shows the wrong fields for the selected value. JS corrects it on load, so it passes any post-load DevTools check; the bug is the first-paint / no-JS state. Fingerprint: a flash of the wrong field group on load, or a fully-wrong form when JS is blocked. **Fix: render the initial `hidden` attributes to match the first/selected option** (default value `brewing` → brewing group NOT hidden, non-brewing group `hidden`); the toggle JS then keeps them in sync on change. General rule: the default-rendered DOM must already BE the correct no-JS state — JS enhances, it never repairs. This is the mirror of the `[hidden]` vs `display` trap above (there CSS overrode the attribute; here the server set the attribute wrong). (2026-06-16, planning.php wort add-form: `.pl-nonbrewing-fields` rendered visible while `brewing` was the default process; added `hidden` to match.)
- **Server-side image render fails in the web context (empty PATH).** php-fpm runs with `clear_env` defaulting to `yes` and `env[PATH]` commented out → web workers have NO PATH. A PHP page calling a bare CLI binary (`exec("pdftoppm …")`) fails (CLI works because the login shell has PATH). Fingerprint: the full-file/download endpoint (no exec) works, but the generated-preview endpoint 502s/blank for anything not already cached. **Fix: call binaries by absolute path** (`/usr/bin/pdftoppm`) in any web-context `exec`/`shell_exec`; and/or **pre-warm the derived cache** (CLI-render all assets) so the web path serves cache-first and needs no exec. (2026-05-25, document_preview.php) (see [[ingest-real-bytes-not-symlinks]], [[vps-document-pdf-archive]])
- **`window.X` payload key-collision.** When the same object carries both a count and an array under near-identical names, a careless build can set the field to the wrong shape. Real case: a supplier object had `stats.invoices` (int count) AND `invoices` (array of headers); if the array key gets overwritten by the count, the JS `(s.invoices || []).length` path silently renders an empty "no items" section → no clickable elements → "popup doesn't work" with no error. Keep count and collection under clearly distinct keys; verify the actual emitted JSON shape, not just the query.
- **Unescaped interpolation in client-rendered HTML = XSS + breakage.** JS that builds HTML from data MUST run every interpolated value through an `escHtml()` helper (and `encodeURIComponent` for URL params). Unescaped supplier names / refs break the markup and open XSS. (Server-side `json_encode` already needs `JSON_HEX_TAG|JSON_HEX_AMP` when embedded in a `<script>`.)
- **Stale render after deploy.** If a change "isn't taking", confirm the cache-bust: the PHP must link assets with `?v=<?= filemtime($absPath) ?>` and the page HTML must be re-fetched. A deployed JS file with an unchanged `?v=` (e.g. mtime not updated, or chowned/copied preserving mtime) serves stale from browser cache. **The `?v=` bust protects JS/CSS only — NOT a change to the PHP markup itself** (a new form field, a chip, a `<div>` you added directly in `*.php`). That depends on PHP-FPM picking up the new bytecode. Our pool runs `opcache.validate_timestamps=On, revalidate_freq=2`, so a single-file deploy is normally live within ~2s — but if a PHP-markup change still doesn't appear: (1) reload php-fpm (`sudo systemctl reload php8.1-fpm`) to force-flush opcache, (2) hard-refresh to drop the browser-cached HTML, and (3) confirm you're actually on the **view/state that renders it** (e.g. a form field gated behind selecting a site won't show on the empty landing state). A CLI `php -l`/`php -r` smoke does NOT exercise FPM's opcache, so it passes while the live page stays stale. (2026-06-12, expeditions stocktake "Clôture mensuelle" checkbox not visible after a surgical single-file deploy — file was correct on disk; fpm reload + hard-refresh resolved it.)
- **Theme-token drift from mockups.** Porting a `public/_design/` mockup that used its own `--paper*`/`--copper*` tokens without remapping to the real palette → undefined vars → invisible text / broken cards. Remap during integration.
- **Dark-background element using the default ink tokens → black-on-black, invisible text.** The app is a LIGHT kraft-paper theme: the `:root` ink tokens (`--ink` #241b10, `--ink-soft`, `--ink-mute`) are DARK, meant for text on the light paper/manila surfaces — and `body.home` does NOT override them. So any element you give a deliberately DARK background (a glass tooltip, a dark display-case card, an overlay chip) must set LIGHT text explicitly — cream `--bg-elev`/`--bg`, muted tan `--ink-faint`, plus a light accent (a mid token like `--bbt` #2f6d99 fails AA on a near-black chip — use a light hex like `#7ec8f4`). If you leave `color` to inherit or use `var(--ink*)`, the text renders dark-on-dark: present in the DOM but unreadable. Fingerprint: the chip/box is visible (its hardcoded dark bg paints) but its text is "completely black"/missing; computed `color` is a dark value. Inverse of the surface-contrast rule (mid-tone-as-text must pass AA on paper) — here the surface is dark, so the text must be light. (2026-06-11, CCT detail-modal "historique" tooltip `045994d`: dark-glass chip pulled `--ink-soft`/`--ink`/`--ink-mute` → black-on-black; fixed to `--bg-elev`/`--bg`/`--ink-faint` + `#7ec8f4` CC row. The data was always there, just invisible.)
- **`--ink-faint` is a DIVIDER tone, NOT a text color → light-on-light, unreadable. For faint TEXT use the dedicated `--ink-label` (#8a7250).** `--ink-faint` (#c8b48a) is the LIGHTEST ink token (a pale tan); on the light paper/manila surfaces (`--bg` #f1e8d4, `--bg-elev` #ece0c6) it fails AA badly as text — present in the DOM, just barely visible. It is structurally **overloaded** (historically used for both faint text AND hairlines/borders/`color-mix` tints/SVG strokes/scrollbars; `--hairline` shares its hex), so you must NOT darken its value to fix text — that wrongly darkens every divider. **The token-split (2026-06-15): `--ink-label` (#8a7250) is THE token for faint/overline/meta/label/caption TEXT — passes AA on paper, still reads as a soft label not body ink; `--ink-faint` is now divider/border/tint/stroke ONLY.** For stronger label text use `--ink-mute` (#4a3820) or `--ink-soft`/`--ink`. **Carve-outs that stay on `--ink-faint`:** text on a deliberately DARK surface (the correct light tan — see the dark-on-dark entry below, the inverse case), and `::placeholder` (a darkened placeholder reads as a filled-in value). Fingerprint: operator says a heading/label is "hard to read"/"too light"; computed `color` resolves to `#c8b48a` against a cream background → it should be `var(--ink-label)`. (2026-06-15: a global sweep added `--ink-label` and repointed ~251 `color: var(--ink-faint)` text uses across 24 CSS files onto it; the token name `--ink-faint` lies — read the surface it sits on.)
- **`.home .main` centers its children → full-bleed / off-canvas / sliding-stage pages render BLANK.** After the "drop sidebar" nav refactor (`9e87a3f`), `.main` is full-width and the global rule `app.css:669` `.home .main` gained `align-items: center` to center normal-width page content. Any page whose `.main` holds a child WIDER than 100% (a `width:400%` sliding stage, a horizontal carousel, an off-canvas panel that relies on being LEFT-anchored at x=0) gets that wide child CENTERED — its left edge computes negative (e.g. a 400%-wide stage in a 1487px viewport lands at x=−2230px), shoving the first/visible panel entirely off-screen left. The viewport then sits over empty lazily-painted detail panels → totally blank page, **zero JS/PHP errors** (the page renders fine, it's just translated out of view). Fingerprint: full clean HTML/CSS/JS, no console errors, but nothing visible; DevTools shows the stage rect at a large negative x. **Fix: the page's own `.main` override MUST explicitly reset `align-items: stretch` (or `flex-start`)** — overriding `.home .main`'s centering — with a comment saying so. A `.main` override that sets `display/flex-direction/height/overflow/padding/position` but omits `align-items` silently inherits the centering. (2026-05-27, Salle des Machines sliding-stage `body.salle-des-machines .main`; proven via Playwright stage rect @(−2230,107)→@(0,107). Audit any page with a `width>100%` flex child under `.main`.) **Second manifestation (subtler, same root cause): a full-WIDTH child fills correctly, but its normal-width siblings DECOUPLE.** A `repeat(7,1fr)` week grid stretches to fill `.main`, while the page's own header / week-nav / flash (normal-width flex children of the same `.main`) get `align-items:center`-aligned and sit in the column centre — so the nav labels float left of where the full-width grid starts, visibly detached, on any screen wider than ~900px. No blank page this time, just a broken-looking layout. Same fix: the page's `.main` override sets `align-items:stretch`. The rule generalises — ANY page whose `.main` holds children of differing widths needs the stretch reset, not only the off-canvas `width>100%` case. (2026-06-16, planning.php week grid: `body.planning-page .main{align-items:stretch}`.)
- **`overflow-x` on the app shell: scroll, don't clip — but full-bleed/sliding-stage pages must self-clip.** `body.home` and `body.auth` are `overflow-x: auto` (app.css, `65f1b93`), NOT `hidden`: content wider than the viewport (narrow window, wide grids/tables/forms, multi-column op-form grids) must produce a horizontal SCROLLBAR, never be cut off unreachably. The fixed `::before`/`::after` wash layers are viewport-bound and add no scroll width, so they don't trigger a spurious scrollbar. **Exemption (must preserve):** `body.home.salle-des-machines { overflow-x: hidden; }` — that page slides a `width:400%` `#sdmStage` via `translateX` (`salle-des-machines.js:46`) and relies on the body clipping the off-screen rooms; without the clip it would show a 400%-wide scrollbar. **Rule:** never re-add `overflow-x:hidden` to `body.home`/`body.auth` globally; any NEW full-bleed/sliding-stage page must add its OWN scoped `overflow-x:hidden` (on its body class) or self-clip its stage container — pair this with the entry above (such a page ALSO needs `align-items` reset on its `.main`). This layers on the 2026-05-31 "fit guard" (`.home .main > * { max-width:100%; min-width:0 }`, which still caps direct children); genuinely-wide content now scrolls instead of being capped/clipped. (2026-06-03, operator-reported "narrow window cuts content everywhere".)

- **Selector-card identity must carry the raw backend match-key, not the display label.** A clickable card that deep-links (sets a GET param or hidden field the backend looks up on) must put the EXACT string the backend matches on into its `data-*`/value — the *raw* source column, not the human/short display name. Real trap: candidate cards for in-tank lots set `data-beer` from the recipe's short name, while the firewall/phase-detection matched on the raw brewday `beer` string; for the contract/collab recipes whose short name ≠ raw name (e.g. `Pale Ale` vs `Le Traquenard - Pale Ale`), the click deep-linked to a not-found state (`Aucune CCT trouvée`) with **zero errors**. Keep display and identity as two separate fields: `beer_display = COALESCE(NULLIF(raw,''), name)` for the label, `data-beer = raw` for the lookup. Core items mask the bug (their short name happens to equal the raw key) — **the test that exercises the fix is a mismatching contract/collab item, never a core beer.** (2026-06-01, fermenting state-gated selector; mirror racking's `data-neb-beer = $cand['beer']`.)

- **SVG element present but invisible → painted BEFORE an opaque sibling (z-order = document order).** SVG has no z-index: later siblings paint ON TOP of earlier ones. An indicator layer (fill-level tint, highlight, hatch) emitted before an opaque body fill is 100% hidden — and the trap is that it LOOKS like a subtlety problem, so the instinct is to crank its opacity, which does nothing because the element is *underneath* an opaque sibling, not faint. Fingerprint: the element exists in the DOM with correct geometry/fill (DevTools shows it), but nothing renders; opacity changes have zero visible effect. **When porting SVG markup between implementations (mockup HTML → PHP function, JS painter → PHP), verify PAINT ORDER against the spec, not just element presence** — a port can emit all the right elements in the wrong sequence and pass any "is it there" check. Fix: emit in layer order (opaque body → tints/indicators → hatching/strokes) and eyeball a rendered screenshot, not the markup. Companion trap from the same incident: a `<text>` label placed near the viewBox edge gets CLIPPED mid-glyph (a "100" renders as "10") — right-anchor it (`text-anchor="end"` at x just inside the edge) instead of left-anchoring and hoping it fits. (2026-06-07, gravure vessel port `9dcd4ca`: fill-tint rect painted before the paper-body polygon — operator reported "BBT fill level unclear"; tint was invisible, not faint; fixed `49981f4`/`c31163f`. The `_design` spec had correct order; the port inverted it.)

- **Dropdown/popover clipped → `position:absolute` inside an `overflow:hidden|auto` ancestor.** The single most common dropdown bug in generated code. Fix preference (in order): 1. Native Popover API (`popovertarget` + `popover` attributes) — renders in the browser top layer, light-dismiss, escapes all clipping, no z-index fight. 2. `position: fixed` + `getBoundingClientRect()` coordinates set via JS. 3. Restructure the ancestors to remove the offending `overflow` constraint. Audit any `position:absolute` dropdown whenever its parent or any ancestor has `overflow` set.

- **PRG submit endpoint that never `require`s its flash/redirect helpers → HTTP 500 on submit, only ever on the form action.** A form-submit API (`public/api/<form>-phase-submit.php`) that follows the post/redirect/get pattern calls `flash_set()`, `redirect_to()`, and `pdo_friendly_error()` — **all three defined in `app/settings-helpers.php`** — but that file is required by the *module page* (`public/modules/form-<x>.php`, which renders the form), NOT by the *API endpoint* the `<form action>` actually POSTs to. The endpoint's own require list (`auth`, `csrf`, `db-write-helpers`, `sessions`, `yeast-eligibility`) does NOT transitively pull settings-helpers. Result: **`Call to undefined function flash_set()` / `redirect_to()` → uncaught Error → 500** on the submit. Fingerprint: the form *renders* perfectly (module page loaded its helpers) but pressing submit yields a bare "This page isn't working / HTTP ERROR 500"; nginx log shows `PHP Fatal error: Call to undefined function flash_set() in …-phase-submit.php`. Insidious because the endpoint can appear "live" for a long time: `flash_set`/`redirect_to` fire on EVERY path including the success flash, so the bug surfaces only the first time a real POST reaches the endpoint — a pilot whose submit was UI-gated (a firewall disabling the button) hides it until the gate is removed. **Fix: the API endpoint must `require_once app/settings-helpers.php` itself** (it owns its PRG-helper dependency; don't rely on the rendering page). **Verification rule: never declare a form done by loading the page — POST it once end-to-end and confirm the redirect+flash, because the render path and the submit path load different files.** (2026-06-03, fermenting-phase-submit.php; exposed when the CIP-cadence firewall gate was removed and operator backlog finally reached submit.)

- **`role="listbox"` on a grid of buttons is invalid ARIA, and `aria-live` on a rebuilt container spams screen readers.** `listbox` requires `option` children — it's right for an actual option list (like the type-ahead picker in Pattern 3 below), wrong for a grid/board of `<button>` cards, where AT announces a broken widget. And putting `aria-live` on a container that JS rebuilds (polling, filtering, re-render on selection) makes AT re-read the ENTIRE contents on every rebuild — unusable noise, not accessibility. **Fix: `role="group"` + `aria-label` on the container, and a SEPARATE sr-only `role="status"` element that the rebuild function writes a short summary string into** ("12 lots disponibles") — one calm announcement per change, buttons stay native. The summary write belongs in the same function that rebuilds the grid (e.g. `buildTankCardsFromData`), so it can never drift from what's shown. (2026-06-07, ui-ux Phase 3 pre-commit review caught this as a P1 in the tank-grid bundle — the "add ARIA" instinct produced something worse than no ARIA.)

- **Never bolt a `preventDefault()` pre-submit gate onto a `novalidate`, server-validated PRG form.** These forms validate server-side by design (`novalidate` is intentional; the server rejects + flashes + redirects, and sticky-on-error re-fills the fields). A client-side submit listener that `preventDefault()`s on its own validation check creates a SECOND validation authority that can silently disagree with the server: a legit submit gets blocked with **no error message and no network request** — the operator presses the button and nothing happens. It also changes flow behavior on a form whose submit path other machinery depends on (autosave clear-markers, double-submit guards). Fingerprint: submit button "does nothing", zero console errors, zero POST in the network tab. **The right altitude for focus-first-invalid a11y is POST-server-validation: on the PRG error redirect, the server knows exactly which fields failed — emit that, and focus the first invalid field on the re-rendered GET.** Client JS may *decorate* (inline hints, counters) but must never *gate* submission on these forms. (2026-06-07, ui-ux Phase 3 pre-commit review: a well-intentioned a11y gate on the packaging form was removed entirely as a P1 flow-change risk; the proper PRG-side fix is logged as its own build.)

- **A tab-switch handler that ENUMERATES known face names mis-routes any NEW tab you add.** When extending an existing tablist with a 3rd/Nth face, check how the switch JS resolves the target panel: if it special-cases the original names (a hardcoded ternary `target === 'stock' ? stockPane : /-activite-\d+$/`, an `if/else` over known keys, a switch with no default), a newly-added tab falls through to the wrong branch — your new panel never shows AND a sibling mis-toggles, reading as "the new tab does nothing / shows the wrong content". The original tabs work, so the tablist looks fine until you add to it; latent until first extended. **Fix: drive the switch generically off the `data-*` attribute** — `const target = btn.dataset.evtTab; panes.forEach(p => p.hidden = (p.id !== 'exp-pane-'+target+'-'+id))` (or a regex built from the escaped target), so any face name works including future ones. When adding a face, also confirm the new `role="tab"`/`role="tabpanel"`/`aria-controls`/`aria-selected`/`hidden` wiring matches the existing faces exactly so the generic switcher (and AT) handle it. (2026-06-11, expeditions stock-drill 3rd "Couverture" face: pre-existing hardcoded-ternary switcher mis-routed it → fixed to a generic regex in `expeditions-stock.js`.)

- **`const`/`let` aliasing a shared global at the top of a classic `<script>` collides with another classic script's top-level binding → the WHOLE file dies at parse, the page goes blank.** Multiple non-module `<script>` tags share ONE global lexical scope. If `page-kpis.js` opens with `const escHtml = window.KpcCharts.escHtml; const fmt = …` while `kpi-charts.js` (loaded earlier on the same page) already declares a top-level `function escHtml`/`const fmt`, the second declaration throws `SyntaxError: Identifier 'escHtml' has already been declared` — and a parse-time SyntaxError discards the ENTIRE file, including unrelated code in it (the tab switcher, the page bootstrap, the chart init). So a page that merely *shares a helper name* goes fully blank/dead with no partial behaviour. Fingerprint: a whole page suddenly "does nothing"/renders blank right after you added or started sharing a helper; the console shows exactly one `Identifier '…' already declared` SyntaxError; backend + `php -l` are clean (it's a browser parse error only, invisible to CLI lint and to data-layer checks). **Fix: alias shared globals with `var`, never `const`/`let`** — `var escHtml = window.KpcCharts.escHtml;` — `var` redeclaration across scripts is legal and idempotent; reserve `const`/`let` for names scoped INSIDE an IIFE/function. Better still, wrap each page's JS in an IIFE so its locals can't collide at module scope at all. Caught only by a real browser console capture, never by `php -l`. (2026-06-15, wort.php blank: `const escHtml`/`fmt` aliases clashed with kpi-charts.js's top-level `function escHtml` → whole file parse-died → tab switcher never ran → blank page; fixed by switching the 6 aliases to `var`, `5561145`.)

## Operator form resilience patterns

### Pattern 1 — Global keepalive (session-idle-expire prevention)

PHP sessions idle-expire (~30 min default). A long-fill operator form submitted after a session drop gets a 302-to-login and all field values are gone. The fix is a **global keepalive + CSRF-sync** wired once for the whole app, not per-form.

How it works in this repo (canonical example: `public/api/session-ping.php` + `public/js/form-resilience.js`, wired in `app/partials/topbar.php`):

- `session-ping.php` is a read-only authenticated touch: it calls `current_user()` (which resets the idle clock) but **NOT** `require_login()` (which would 302-redirect, breaking `fetch()`). On true expiry returns `401 JSON`, never a redirect — that's the signal the client consumes.
- `form-resilience.js` runs globally (loaded by topbar on every authenticated page). It pings every 5 minutes **while the tab is visible** (`document.hidden` check via Page Visibility API). On tab becoming visible again, it pings immediately + re-arms the timer.
- On a successful ping, the response includes a fresh `csrf` token. The keepalive **rewrites every `input[name="csrf"]` on the page** from it. This catches CSRF invalidation due to session-ID rotation (happens every ~15 min) or session rebuild from a `mt_remember` cookie — without the rewrite, the next POST would fail with `{ok:false,reason:'expired'}`.
- On 401 (truly gone, no remember-me): a non-destructive sticky banner tells the operator to open a new tab and re-login; the form is **NOT reloaded** (which would wipe unsaved fields).

Reuse: wire once in `topbar.php` and it covers every authenticated page automatically. No per-form changes needed.

### Pattern 2 — Autosave for batch forms (localStorage draft)

For forms where the operator fills many fields before submitting (classic PRG batch form — NOT the line-append archetype below), `FormResilience.initAutosave({ formSelector, storageKey })` serialises all named non-CSRF inputs to localStorage on every `input`/`change` event. On page load, if a draft exists, values are restored and a dismissable banner is shown. The draft is cleared **only on confirmed success**: the form sets `sessionStorage['form_just_submitted'] = storageKey` on submit; on the redirected GET, if an ok-flash banner is present AND the marker matches, the localStorage draft is purged. This is the most reliable signal — PRG means a redirect happened, and the ok flash confirms server success. Never clear the draft on the submit click itself (the server may still reject it — CSRF fail, validation error).

The `storageKey` should include any period or context that scopes the form (e.g. `'form-rm-stocktake:2026-05'`) so drafts don't bleed across months or between different operator sessions.

**Note:** the RM stocktake form does NOT use `initAutosave` — its line-append archetype (see below) supersedes it because every datapoint is already persisted server-side on entry.

**Derived/read-only fields must be recomputed AFTER a programmatic restore — not just on user events.** Draft-restore (and edit-mode prefill, and browser autofill) repopulate inputs by setting `el.value` directly, which does **NOT** fire `input`/`change`. So any field that is *derived* from other inputs via a change-listener (a read-only count, an auto-sum, a computed total) silently stays stale/zero after a restore even though the source values are visibly back. Fingerprint: "I refreshed, my inputs came back, but the computed field shows 0/old." (2026-06-05, packaging form: `qa_analyses_units` derived from the count of CO₂/O₂ in-filling read pairs — `syncQaAnalysesDisplay()` was only called on add/remove-row, on the MSR widget `onChange`, and inside the *edit-mode* block; FormFramework's `loadDraft` set values without dispatching events, so a plain draft-restore never re-derived it.) **Fix: call the derivation once immediately after the restore runs** (`loadDraft` is synchronous inside `FormFramework.init()`, so a single call right after `init()` works), AND wire a delegated `input` listener on the source container as belt-and-suspenders so any value path re-syncs. General rule: a derived field needs a recompute trigger for *every* way its inputs can change — user typing, dynamic add/remove, programmatic restore/prefill — not only the first one you tested.

### Pattern 3 — Per-item line-append entry form (instant-persist, no batch submit)

The canonical example is the RM stocktake form (`form-rm-stocktake.php` / `form-rm-stocktake.js`). This is the right archetype when the domain has **multiple independent line items** (pallets, entries, receipts) that should each persist immediately, not accumulate in a local buffer for a single batch submit. Contrast with a classic form where the whole state is submitted at once.

Anatomy:

- **Server renders data via `window.X` injection.** The PHP page (`GET` only — no POST handler) builds the list of valid options (`window.RM_MIS`), the session CSRF (`window.RMS_CSRF`), and the current period (`window.RMS_PERIOD`) using `json_encode(..., JSON_HEX_TAG | JSON_HEX_AMP)`. The JS hydrates from these; no AJAX call on initial load.
- **Type-ahead picker over `window.X`.** A plain text input filters the injected list (`includes()` on both name and code); results rendered as a `<ul role="listbox">` with arrow-key navigation. Once selected, the search input hides and a summary chip shows with a clear (✕) button. This is client-side filtering of an injected list — not an AJAX search endpoint.
- **Instant per-item POST on "Add".** Each entry triggers a single JSON endpoint (`rm-stocktake-line-add.php`) that writes the line and returns the updated subtotal + grand total. The JS applies the response to the DOM live: adds a chip to the ledger, updates the per-ingredient subtotal, updates the grand total. No page reload.
- **Live ledger DOM update.** Groups are keyed by `data-mi-id`; chips keyed by `data-line-id`. On add: find-or-create the group, append a chip, update subtotals. On delete: remove the chip, re-update subtotals, remove the group if now empty. The server response carries the authoritative `mi_subtotal` and `grand_total` — the JS trusts those numbers rather than recomputing client-side from partial DOM state.
- **No batch submit button.** The "Add" button IS the save action; every entry lands server-side immediately. The only fixed/floating element is a back-link (`← Retour`) that the operator clicks when done. This makes the form inherently loss-proof: a browser crash between two entries at most loses the one item being typed, not the whole session.
- **CSRF refresh-and-retry (once).** Both add and delete endpoints return `{ok:false, reason:'expired', csrf:<fresh>}` on CSRF mismatch. The JS local CSRF variable is updated from the keepalive ping (via `updateAllCsrfInputs()`). On `{reason:'expired'}`, the JS refreshes its local token from the response and retries the POST exactly once. Combined with the global keepalive rewriting `input[name="csrf"]` hidden fields, this makes CSRF expiry seamlessly transparent to the operator.
- **RM stocktake form is now per-pallet multi-line (line-append + rollup).** Do NOT reintroduce the retired single-summed-input model (one counted_qty text field, one submit) for RM stocktake. Brewing/racking/packaging forms remain single-summed-input for now.

## "Popup / preview / page doesn't render" — debugging playbook

Work the layers in order; each step bisects the problem. **Network-first**: the single most diagnostic fact is whether the request even leaves the browser.

1. **Does the request reach the server?** `ssh maltyweb 'sudo grep -h "<endpoint>" /var/log/nginx/access.log | tail'`. 
   - **Hits with 2xx** → server is fine; the bug is in *displaying* the response (CSS visibility, wrong content-type, JS not inserting it).
   - **Hits with 4xx/5xx** → server-side (auth 403, missing file 404, exec/render 502). Reproduce server-side.
   - **ZERO hits** → the browser never sent it → the bug is client-side (JS didn't run the fetch/set `img.src`, or the element is a hidden lazy image). This is the highest-signal outcome — stop looking at the server.
2. **Isolate server vs client with a direct hit.** Open the media/endpoint URL directly in a logged-in tab (`/api/document-preview-png.php?file_id=…`). Works directly but not in-page → purely a client/markup bug (e.g. the lazy-hidden trap, or the element never created). Fails directly → server/endpoint bug.
3. **Confirm the data shape.** Re-run the page's data query (`sql` skill) and check the *emitted* `window.X` JSON shape — empty arrays, key collisions, `has_*` flags false, null ids → the section renders empty / elements aren't clickable.
4. **Confirm the deployed asset.** `grep` the live `public/js|css` file for the expected code; confirm `?v=` mtime changed after deploy. Rule out browser/SW stale state with an "Empty Cache and Hard Reload".
5. **Then DevTools.** Console (red errors abort the render path before later code runs — e.g. a throw between building a modal and setting `img.src`), Network (status of the sub-resource), Elements (is the node present but `display:none` / zero-size / behind another layer / wrong color-on-color).

A throw partway through a render function leaves a half-built UI: earlier DOM (and `<a href>` links) work, but everything after the throw (e.g. `img.src=`) never runs — which looks exactly like "the modal opens and the PDF link works but the image is stuck."

## Verification before declaring a UI change done

**If any box below cannot be honestly ticked, the surface is not done — fix before delivering.**
Models trained for brevity treat "prefer"-phrased rules as optional; binary gates and named
banned patterns get complied with. The catalog entries above name the exact pattern/string to
look for; use them.

- **Actually render it in a browser** (preview URL or the live module) and exercise the golden path AND the edge cases. Type-check/lint proves code validity, NOT that the feature renders. If you cannot open a browser, say so explicitly — do not claim the UI works.
- For a modal/viewer: open it, confirm the on-demand asset loads (watch the Network tab / access log), close it (✕, backdrop, Escape), reopen (no leak/dup), and confirm it works on a not-yet-cached item (not just the one you pre-warmed).
- Confirm CSS is scoped under the body class and uses real palette tokens; no inline `<style>`/`<script>`; values are `escHtml`/`encodeURIComponent`-escaped.
- Confirm the cache-bust `?v=` advanced after deploy.
- **Post-deploy HTTP smoke (the "500 after deploy" class).** After `bin/deploy.sh`, `curl -s -o /dev/null -w '%{http_code}' https://<host>/modules/<page>.php` for the pages you touched — expect `200/302/401/403`, never `500`. A PHP fatal / blank page on first load won't show in your local preview. Wire the multi-page battery into the deploy itself — see `coder` "Pre-deploy gate, smoke test & rollback".
- No DB codes leaked into operator-facing text (label-map applied).
- `git status` shows only intended files; no stray `_design` experiments left committed unless intended.
- **UX-quality sweep**: run the §9 checklist in `references/ux-quality-rules.md` (contrast, keyboard, aria, touch targets, async-button locking, locale formatting, chart states).
